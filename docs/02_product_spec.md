# 02 软件说明文档 / Product Specification

## 1. 文档目的

本文档是项目最重要的需求来源。

所有功能设计、技术实现、验收测试和开发计划都应该优先参考本文档。

如果本文档与临时对话内容冲突，应先提醒用户确认，再修改文档或代码。

## 2. 软件名称

Nvy。

## 3. 一句话描述

这是一个面向成年个人用户的 Windows 本地桌面软件，用于通过番号和女优名匹配成人影片及女优基础资料，并记录个人影评、影片时间点感受、女优评价和本地资料库。

## 4. 背景与问题

### 4.1 背景

用户在第三方成人网站浏览和观看影片时，常常会对某些影片、女优或影片中的具体时间点产生个人记录需求。现有网站通常提供观看、搜索和收藏能力，但不提供足够私人化的影评、时间点标注、女优评价和本地资料库管理能力。

用户希望该软件作为独立 Windows 桌面工具存在。观看和挑选影片仍在 MissAV 等第三方网站完成，本软件只负责保存用户自己的影评、女优评价、影片资料和后续检索推荐。

### 4.2 要解决的核心问题

用户观看过影片后容易忘记观看时间、个人感受、值得回看的具体时间点，以及不同女优和影片之间的关联。本软件需要把这些信息结构化保存在本地，让用户可以长期回顾和检索。

### 4.3 当前替代方案

| 替代方案 | 用户如何使用 | 优点 | 不足 |
|---|---|---|---|
| 成人网站收藏夹 | 用户在网站内收藏影片 | 使用方便，与观看站点关联 | 只保存收藏关系，不能记录细致影评、时间点和女优个人评价 |
| 浏览器书签 | 用户保存影片页面链接 | 简单通用 | 缺少结构化字段、影评、时间戳和账号隔离 |
| 普通笔记软件 | 用户手动记录影片和感受 | 灵活 | 难以关联影片、女优、封面、时间戳和后续推荐 |

### 4.4 本软件的独特价值

本软件聚焦成人影片观看后的私人资料整理：影片、女优、影评、时间戳和账号隔离都围绕该场景设计。它不承担播放和下载功能，而是作为本地、隐私优先的记录工具。

## 5. 目标用户

详见 `docs/01_user_profile.md`。

当前核心目标用户：

项目创建者本人，以及具有类似成人影片私人记录需求的成年 Windows 用户。

## 6. 使用场景

### 6.1 核心使用场景

用户在 MissAV 等网站完成影片浏览和观看后，打开本软件记录影片资料、撰写影评、插入时间点，并将影片关联到女优。

### 6.2 次要使用场景

- 用户查看某位女优的资料、个人评价和已记录影片。
- 用户查看首页轮播的收藏女优、收藏影片并跳转详情。
- 用户导入或导出本地数据备份。
- 第一版首页保留对话框，并实际调用大模型与用户交互：根据首页下方文本框输入检索本地资料库，推荐影片和女优。第一版还需要调用大模型自动翻译影片标题和简介，翻译结果保存到本地，避免重复翻译。

### 6.3 不支持的使用场景

- 不在软件内播放成人影片。
- 不下载、托管或分发成人影片。
- 不绕过第三方网站访问控制。
- 不自动抓取第三方网站数据，除非后续明确确认合法、稳定、必要。
- 不处理未成年人相关内容。

## 7. 第一版 MVP 范围

第一版 MVP 只实现最小可行功能，用于验证软件核心价值。

### 7.1 第一版必须实现

| 编号 | 功能 | 为什么必须有 |
|---|---|---|
| M001 | 本地账号注册与登录 | 需要支持多个本地账号，并隔离不同账号的数据 |
| M002 | 影片资料管理 | 软件核心对象是成人影片记录 |
| M003 | 影片影评管理 | 解决用户记录观后感的核心痛点 |
| M004 | 影评时间戳插入 | 解决用户记录具体影片时间点的核心痛点 |
| M005 | 女优资料管理 | 用户希望围绕女优建立评价和关联影片 |
| M006 | 影片与女优关联 | 支持从女优详情查看其相关影片 |
| M007 | 本地自动保存 | 用户不希望手动保存，且记录不能丢失 |
| M008 | 数据导入导出 | 第一版必须支持备份、迁移和恢复本地资料库 |
| M009 | 影片元数据匹配 | 用户只输入番号时，应尝试匹配影片封面和基础资料，降低录入成本 |
| M010 | 女优元数据匹配 | 用户只输入女优名时，应尝试匹配头像和女优基础资料，降低录入成本 |
| M011 | 首页对话框样式与大模型配置入口 | 第一版需要预留首页对话入口样式，并在设置页保存 API、模型参数和提示词配置 |
| M012 | 图片本地缓存 | 匹配成功后的封面和头像需要本地保存，避免外链失效 |
| M013 | 删除二次确认 | 影片和女优记录误删成本高，需要确认机制 |
| M014 | 作品类型与女优关联规则 | 需要根据单人、多人、素人作品分别处理女优必填、自动关联和后续补关联 |
| M015 | 标签库与组合搜索 | 用户需要通过标签、关键词、作品类型和排序组合检索本地资料库 |
| M016 | 女优多语种姓名与显示规则 | 用户需要用中文、繁体、日文、罗马音和大小写不敏感方式查找同一女优 |
| M017 | 中文标题和中文简介 | 自动检索到的影片标题和简介可能是日文，第一版需要解决中文化显示问题 |
| M018 | 大模型设置与提示词管理 | 用户已确认会接入大模型，第一版需要保存 API、模型参数、翻译提示词和推荐提示词 |

### 7.2 第一版可以延后

| 编号 | 功能 | 延后原因 |
|---|---|---|
| L001 | 在线服务器同步 | 第一版先验证本地记录闭环，服务器会显著增加复杂度 |
| L002 | 云端账号体系 | 第一版使用本地账号即可满足数据隔离 |
| L003 | 更复杂的大模型记忆和多轮策略 | 第一版只做基于本地候选资料的推荐和标题/简介翻译，复杂模型记忆、工具调用和多轮策略后续优化 |
| L004 | 安装包发布 | 第一版只供项目创建者本人使用，稳定后再考虑打包成安装包 |

### 7.3 第一版明确不做

| 编号 | 功能 / 能力 | 不做原因 |
|---|---|---|
| N001 | 影片播放 | 软件定位是记录工具，不是播放器 |
| N002 | 影片下载 | 涉及版权、合规和复杂技术问题 |
| N003 | 第三方网站深度集成 | 第一版只做元数据匹配，不做播放、下载或站内操作集成 |
| N004 | 在线多人共享 | 与私人本地记录定位不一致，且会增加隐私风险 |
| N005 | 未经确认的数据发送 | 在用户确认发送范围前，不把本地私人资料发送给大模型服务 |

## 9. 用户操作流程

### 9.1 主流程

1. 用户打开软件。
2. 用户在登录界面选择离线使用。
3. 用户注册或登录本地账号。
4. 用户进入首页，通过左侧栏进入“影片”或“女优”。
5. 用户新增或打开一部影片。
6. 用户输入番号，软件尝试匹配影片基础资料和封面。
7. 软件展示匹配候选结果，用户选择一个结果或跳过匹配。
8. 用户检查并自由修改影片资料。
9. 用户选择影片类型：单人作品、多人作品或素人。
10. 用户撰写影评。
11. 用户在影评编辑时输入时、分、秒并点击添加，软件在当前光标处插入 `[01:23:45]` 格式时间戳。
12. 如果影片是单人作品，用户必须输入女优名，软件尝试匹配女优头像和基础资料。
13. 软件展示匹配候选结果，用户选择一个结果或跳过匹配。
14. 用户检查并自由修改女优资料。
15. 如果影片是单人作品，软件关联该女优；如果影片是多人作品，软件根据演员名单自动关联已收藏女优，未收藏女优暂不关联，并在以后添加匹配女优时弹出确认后补关联；如果影片是素人作品，允许用户手动关联已收藏女优。
16. 软件自动保存数据。
17. 用户可导入或导出本地数据。
18. 用户后续可通过影片详情或女优详情查看记录。

### 9.2 异常流程

- 用户名为空、长度不符合要求或包含不支持字符时，提示用户修改。
- 密码为空、长度不符合要求或包含非字母数字字符时，提示用户修改。
- 注册时用户名已存在，提示用户更换用户名或登录。
- 登录密码错误，提示账号或密码不正确。
- 影片番号为空时，提示用户补充必要信息。
- 影片元数据匹配失败时，提示用户未找到匹配结果，并允许手动创建或编辑。
- 女优元数据匹配失败时，提示用户未找到匹配结果，并允许手动创建或编辑。
- 元数据匹配出现多个候选结果时，默认展示候选列表，由用户确认，不自动静默覆盖现有资料。
- 单人作品未填写女优名时，拒绝保存或提示补充女优名。
- 多人作品导入演员名单时，仅关联已收藏女优；未收藏女优不创建新女优记录。
- 后续新增女优时，如果其姓名或别名匹配既有多人作品演员名单，软件应弹出确认，由用户确认后补充关联。
- 素人作品不强制填写女优，但允许用户手动关联已收藏女优。
- 删除影片或女优时，必须进行二次确认。
- 删除女优时，不删除该女优曾关联的影片。
- 时间戳输入不是有效时分秒时，拒绝插入并提示格式错误。
- 数据保存失败时，提示用户保存失败，并尽可能保留当前编辑内容。
- API Key 未配置时，大模型推荐和翻译不可用，并提示用户前往设置。
- 网络不可用时，所有本地记录功能仍应可用。

### 10.1 影片列表搜索与排序

- 可选排序：按照发行日期排序。
- 作品类型筛选：支持单人作品、多人作品、素人。

- 示例：`#巨乳+#人妻+我` 表示影片标签包含 `#巨乳` 和 `#人妻`，并且出演女优或标题中包含“我”。
- 年龄标签检索支持：
  - `#[22-33]`：检索拍摄作品时女优年龄在 22 到 33 岁之间，包含 22 和 33。
  - `#[22-]`：检索 22 岁及以上，包含 22。
  - `#[25]`：检索 25 岁。
  - `#[-25]`：检索 25 岁及以下，包含 25。

### 10.2 女优列表搜索与显示

- 女优列表默认展示所有已收藏女优。
- 搜索框支持关键词组合检索，多个关键词用 `+` 连接。
- 搜索标签时必须以 `#` 开头。
- 女优名字搜索需要兼容简体中文、曾用中文名、繁体中文、日文名、罗马音和大小写差异。
- 搜索结果按照匹配相似度从高到低排序。
- 罩杯筛选支持多选，范围为 A-K，排除 I 和 J，并支持全选。
- 右上角可选择名字显示类型：默认、日文名、简体中文名、繁体中文名、罗马音。
- 每个女优可单独配置默认显示的名字类型。
- 如果当前选择的名字类型不存在，按“简体中文 -> 繁体中文 -> 日文 -> 罗马音”的顺序回退显示。
- 默认显示简体中文名字。

#### 女优姓名归一化规则

第一版已确认使用自建规则，不依赖第三方库或外部数据源作为归一化核心。外部元数据来源可以提供姓名、别名和罗马音，但最终是否视为同一女优由本地规则和用户确认决定。

第一版自建规则至少包括：

- 繁简归一：简体中文名和繁体中文名应能互相匹配。
- 大小写归一：罗马音和英文字符大小写不敏感。
- 全角 / 半角归一：全角英数字、半角英数字应统一比较。
- 空格和常见分隔符归一：姓名中的空格、点号、连字符等差异不应导致无法匹配。
- 多字段别名匹配：主名称、简体中文名、繁体中文名、日文名、罗马音名、曾用中文名、其他别名都参与匹配。
- 手动确认兜底：当匹配结果不唯一或置信度不足时，必须展示候选，由用户确认是否为同一女优，不允许静默合并。
- 手动编辑兜底：用户可以在女优详情页自由编辑各语言姓名和别名。

示例：`三上悠亚`、`三上悠亞`、`三上悠亜`、`Yua Mikami`、`yua mikami` 在规则和别名数据充分时应能识别为同一女优候选。

### 10.3 标签库规则

- 影片标签库和女优标签库互相独立。
- 两个标签库都允许用户新增、删除、修改标签。
- 标签原则上必须使用中文；用户已确认的固定缩写 `#NTR` 作为例外保留在预置标签库中。
- 标签应面向成年人合法内容记录，不允许用于未成年人相关内容。
- 影片标签可由用户手动添加，也可由自动规则添加。
- 女优标签由用户自由维护。
- 标签库需要支持规范标签、同义词和关联词。搜索任一同义词或关联词时，应能检索到使用规范标签或相关标签的记录。
- 简体中文和繁体中文差异应被归一化。例如搜索 `#媚藥` 应能匹配 `#媚药`。
- 语义相近标签应支持交叉检索。例如搜索 `#春药` 应能匹配 `#媚药`，搜索 `#精油` 应能关联检索到 `#按摩`，具体关联关系由标签库维护。
- 预置影片标签库：`#按摩`、`#NTR`、`#巨乳`、`#贫乳`、`#精油`、`#人妻`、`#乱伦`、`#熟女`。
- 预置女优标签库：`#苗条`、`#强气`。
- 预置标签中重复项只保留一份；当前影片标签库中 `#按摩` 只创建一个规范标签。

### 10.4 自动标签规则

- 单人作品中，如果女优生日和影片发行日期都存在，软件自动计算发行时女优年龄，并给影片添加年龄标签，例如 `#[22]`。
- 如果女优生日或影片发行日期任意缺失，不自动添加年龄标签。
- 单人作品中，如果女优罩杯存在，软件自动给影片添加 `#xxCup` 标签。
- 自动标签仍应允许用户后续编辑或删除。

### 10.5 大模型设置与提示词

第一版设置页需要保留以下大模型配置项：

- API Key。
- 接口类型：OpenAI Responses API、OpenAI 兼容 Chat Completions、自定义接口。
- Base URL / API 地址。
- 模型型号 / 模型 ID，例如 `gpt-5.5`、`deepseek-v4-pro`；该字段允许用户自由填写，不由软件限制具体型号是否真实存在。
- 供应商名称或备注。
- 温度。
- 最大输出长度。
- 每次推荐参考的女优和影片总数上限：默认值为 30；只能填写 0-999 的数字；填写 30 表示每次发送给大模型的影片和女优数据总数不超过 30 条；填写 0 表示不限数量，全部发送给大模型。
- 首页推荐默认启用：只要 API 配置有效，首页下方文本框就可调用大模型进行推荐。
- 是否启用大模型翻译。
- 首页推荐提示词。
- 翻译提示词。

API Key 属于敏感信息，不应出现在日志、导出明文说明或代码仓库中。第一版需要对 API Key 做本地加密保存。

大模型推荐会把用户输入、候选影片/女优元数据和用户写的文本全文发送给第三方服务。大模型翻译会把待翻译的原始标题和简介发送给第三方服务。软件需要在大模型设置中明确这一点。

推荐功能定义：用户在首页下方文本框输入自然语言偏好，软件从本地数据库检索候选影片和女优，按设置中的总数上限选择要参考的数据，然后将元数据和用户写的文本全文发送给大模型，由大模型返回推荐影片、推荐女优和推荐理由。推荐功能默认开启；当 API Key 或模型配置缺失时，推荐不可用并提示前往设置。

翻译功能定义：当大模型翻译启用时，新增或匹配影片后，如果存在原始标题或原始简介且中文标题 / 中文简介为空，软件自动调用大模型翻译。翻译结果写入本地 `title` 和 `summary` 字段；原文保留在 `original_title` 和 `original_summary` 字段。已翻译且未被用户要求重新翻译的内容，不重复调用大模型。

OpenAI 接口说明：设置中需要支持 OpenAI Responses API。OpenAI 官方文档将 Responses API 描述为用于生成模型响应的接口，请求可包含 `model`、`input`、`instructions`、`max_output_tokens`、`temperature`、`stream`、`tools` 等字段，并通过 `POST /v1/responses` 创建响应。设置中还需要保留 OpenAI 兼容接口和自定义接口，以便切换其他服务。

#### 首页推荐发送数据范围

已确认方案：

| 方案 | 发送内容 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|---|
| A. 最小元数据 | 番号、标题、作品类型、标签、女优名 | 隐私风险最低 | 推荐质量较弱，无法利用个人影评偏好 | 备选 |
| B. 元数据 + 用户摘要 | 番号、标题、简介、作品类型、标签、女优名、女优标签、用户影评摘要、女优评价摘要 | 推荐质量和隐私之间较平衡 | 不能完整利用用户长文本偏好 | 备选 |
| C. 元数据 + 用户文本全文 | 候选影片/女优的元数据、完整影评、完整女优评价、完整简介 | 推荐可参考最完整的本地资料 | 隐私发送范围更大，token 成本更高 | 已确认 |
| D. 全库发送 | 当前账号全部影片和女优完整资料 | 实现简单 | 隐私、成本、上下文长度都不可控 | 禁止 |

第一版采用方案 C。软件先在本地根据用户输入做关键词、标签、女优、作品类型等候选检索，再按设置中的“每次推荐参考的女优和影片总数上限”选择要发送的数据。

数量上限规则：

- 设置项名称：每次推荐参考的女优和影片总数不超过 xx 条。
- 默认值：30。
- 输入限制：只能填写 0-999 的数字。
- 示例：填写 30 时，每次发送给大模型的影片和女优数据总数不超过 30 条。
- 填写 0 时，不限制数量，所有本地检索到的候选影片和女优都可发送给大模型。
- 当检索结果超过上限时，软件需要先按本地匹配度排序，再截取前 xx 条。

发送字段：

- 用户输入原文。
- 候选影片：内部 id、番号、中文标题、原始标题、中文简介、原始简介、作品类型、标签、出演女优名、发行日期、片长、来源链接、用户完整影评。
- 候选女优：内部 id、显示名、简体中文名、繁体中文名、日文名、罗马音名、曾用名、标签、罩杯、生日、身高、三围、出道日期、维基百科链接、用户完整评价。
- 当前筛选条件：作品类型、标签条件、年龄标签条件、罩杯筛选条件。

不发送内容：

- 本地账号密码或密码哈希。
- API Key。
- 数据库路径、本地文件绝对路径。
- 非候选影片/女优的资料；当上限为 0 时，所有本地检索到的候选都可能被发送。
- 未成年人相关内容。

#### 自动翻译触发选项

可选方案：

| 方案 | 行为 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|---|
| A. 自动翻译缺失中文字段 | 新增/匹配影片后，原文存在且中文标题或中文简介为空时自动翻译 | 使用成本低，符合用户希望自动中文化 | 需要处理失败和调用成本 | 推荐 |
| B. 手动翻译 | 只在用户点击翻译按钮时调用 | 最可控、最省 API | 录入流程多一步 | 备选 |
| C. 批量翻译 | 用户在列表中批量翻译缺失中文字段 | 适合整理旧资料 | 需要进度、失败重试和成本提示 | 后续增强 |

第一版采用方案 A，并在影片资料详情页提供：

- `重新翻译` 按钮：重新调用大模型，覆盖当前中文标题和中文简介，但保留原始标题和原始简介。
- `取消翻译` 按钮：清空当前中文标题和中文简介，回到显示原始标题和原始简介的状态；不删除原始字段。

已确认：`取消翻译` 不需要撤销正在进行中的翻译请求，第一版定义为“清空已保存翻译结果”；如果请求稍后返回，不应把已取消的结果重新写入当前影片记录。

#### API Key 本地加密保存方案选项

可选方案：

| 方案 | 行为 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|---|
| A. Tauri Stronghold | 使用 Tauri 官方 Stronghold 插件保存 API Key 等敏感信息 | 与 Tauri 技术栈一致，适合保存密钥，避免明文落盘 | 需要设计解锁方式、备份迁移和多账号隔离 | 推荐 |
| B. Windows 凭据管理器 / 系统 Keychain | 使用操作系统凭据能力保存 API Key | Windows 原生，安全边界清晰 | 跨平台抽象和导入导出迁移更复杂 | 备选 |
| C. 账号密码派生密钥 + AES-GCM | 用本地账号密码派生加密密钥，加密 API Key 后存入 SQLite | 与本地账号绑定，导出迁移逻辑可控 | 自己承担更多密码学实现和忘记密码恢复复杂度 | 后续可评估 |
| D. 明文保存 | 直接保存在 SQLite 或配置文件 | 实现最简单 | 不符合安全要求 | 禁止 |

已确认：第一版采用方案 A，使用 Tauri Stronghold 保存 API Key。SQLite 中只保存非敏感模型配置，例如接口类型、Base URL、模型型号 / 模型 ID、温度、最大输出长度和提示词。API Key 按账号隔离存储，导出时不导出明文 API Key；如后续需要迁移设备，再设计“重新填写 API Key”或“受密码保护的密钥导出”。

#### 元数据与用户文本说明

在首页推荐中，“元数据”和“用户文本”含义如下：

- 元数据：描述影片或女优的结构化资料，不是用户写的观后感。例如影片番号、中文标题、简介、作品类型、标签、出演女优、发行日期；女优显示名、别名、标签、罩杯、生日等。

示例：

```json
{
  "video": {
    "code": "ABCD-123",
    "title": "中文标题",
    "work_type": "单人作品",
    "tags": ["#巨乳", "#人妻"],
    "actresses": ["女优A"],
    "review": "用户影评全文"
  }

```

已确认：首页推荐采用“元数据 + 用户文本全文”的发送方案，并通过“每次推荐参考的女优和影片总数不超过 xx 条”控制发送数量。

### 11.4 本地文件与导出包结构

推荐本地数据目录：使用 Tauri `appDataDir()` / `appLocalDataDir()` 下的应用专属目录。Tauri 官方 path API 说明 `appDataDir()` 返回应用数据文件建议目录，并以 bundle identifier 组成应用目录；`appLocalDataDir()` 返回本地数据目录。

推荐目录结构：

```text
Nvy/
  data/
    nvy.sqlite
  secrets/
    stronghold.dat
  cache/
    covers/
      {video_id}.{ext}
    actresses/
      {actress_id}.{ext}
    tmp/
  exports/
```

图片命名规则：

- 影片封面：`covers/{video_id}.{ext}`。
- 女优头像：`actresses/{actress_id}.{ext}`。
- 不使用女优名、番号或标题作为主文件名，避免重命名、重码和隐私泄露问题。
- 数据库中保存相对路径，不保存机器相关绝对路径。

导出包推荐格式：

```text
NvyExport-{username_or_account_id}-{yyyyMMdd-HHmmss}.nvyzip
  manifest.json
  data.json
  assets/
    covers/
    actresses/
  checksums.json
```

说明：

- 普通导出：`.nvyzip`，本质为 zip 包，包含结构化 JSON 数据和图片资源。
- 加密导出：`.nvyenc`，对完整导出包进行加密。
- `manifest.json` 保存导出版本、应用版本、导出时间、账号 id、数据 schema 版本。
- `data.json` 保存当前账号的影片、女优、标签、影评、设置等可迁移数据；不导出明文 API Key。
- `checksums.json` 保存资源文件校验信息，用于导入时发现损坏文件。
- 导入时先校验 manifest 和 checksums，再写入数据库；导入失败不得破坏现有数据。

### 11.3 数据存储方式

第一版已确认需要本地保存数据。数据存储采用 SQLite；封面、头像等资源保存到本地文件缓存；API Key 使用 Tauri Stronghold 保存。

候选方案：

- 本地 SQLite：适合账号、影片、女优和关联关系等结构化数据。
- 第一版不整体加密数据库：降低实现复杂度，保留未来整体加密升级空间。
- 密码哈希：本地账号密码不明文保存。
- 可选加密导出：导出文件可选择加密，保护备份文件。
- 本地 JSON：简单，但长期维护和查询能力弱。
- 本地文件夹：适合保存封面、头像、导出文件等资源。
- 导入导出文件：第一版必须支持完整导出和恢复本地资料库。
- 图片缓存：匹配成功后的影片封面和女优头像应缓存到本地，避免外部图片链接失效影响浏览。

在线服务器和云数据库作为后续版本方向，不进入第一版。

## 13. 安全与隐私要求

- 本软件保存成人内容相关的私人记录，属于高隐私数据。
- 第一版数据应保存在本地，不上传云端。
- 不同本地账号之间必须隔离数据。
- 密码不应明文保存。
- 第一版本地数据库不做整体加密，但需要保留后续迁移到整体加密数据库的可能。
- 数据导出支持可选加密；加密导出文件的密码或密钥丢失后，导出文件可以无法恢复。
- 用户自填 API Key 应按敏感信息处理。
- 调用大模型 API 时，需要明确哪些本地数据会被发送给第三方服务；首页推荐会发送用户输入、候选影片/女优元数据和用户写的文本全文，自动翻译会发送待翻译标题和简介。
- 大模型 API Key 需要本地加密保存，不应明文保存。
- 需要在文档中明确：软件只面向成年人合法内容记录，不处理未成年人相关内容。
- 元数据匹配需要联网访问第三方数据源，相关数据源的可用性、合法性、速率限制和许可要求需要单独评估。
- 影片封面和女优头像匹配成功后缓存到本地。

## 14. 错误处理要求

错误提示文案指用户操作失败或输入不合法时，界面显示给用户看的提示文字，例如“账号或密码错误”“API Key 未配置”“导入文件损坏”“网络连接失败，请稍后重试”。它不是程序内部异常堆栈，也不应该暴露数据库路径、API Key 或底层技术细节。

具体错误提示文案由实现时根据错误原因编写，不需要逐条预先确认；第一版应遵循以下原则：

错误处理原则：

1. 不应该让用户看到难以理解的底层错误。
2. 软件应该提示用户发生了什么。
3. 软件应该提示用户可以怎么处理。
4. 关键数据不能因为错误而丢失。
5. 对可恢复错误，应提供重试或返回操作。
