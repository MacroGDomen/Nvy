# 03 技术栈选择

## 1. 文档目的

本文档用于记录项目的技术栈选择、选择理由、备选方案、放弃原因、复杂度影响和风险。

技术栈不是越先进越好。本项目优先考虑：

1. 是否适合当前软件目标。
2. 是否适合第一版 MVP。
3. 是否适合项目创建者理解和维护。
4. 是否适合 Coding Agent 稳定生成和修改代码。
5. 是否有足够成熟的文档和社区。
6. 是否会引入不必要的复杂度。

## 2. 当前结论

当前已确认第一版技术栈：Tauri + React + TypeScript + SQLite。第一版应是 Windows 本地桌面软件，数据本地保存，支持本地账号隔离、数据导入导出、影片和女优元数据匹配，暂不实现在线服务器同步。大模型首页推荐、标题/简介自动翻译、模型参数设置、提示词设置和 API Key 本地加密保存进入第一版。

| 层级 | 当前选择 | 状态 |
|---|---|---|
| 软件形态 | Windows 桌面软件，第一版支持 Windows 11 和 Windows 10 | 已确认 |
| 前端 | React + TypeScript | 已确认 |
| UI / 样式策略 | Tailwind CSS + Radix UI primitives + CSS 变量；自定义暗色视觉系统，参考 Google Gemini / Google AI Studio；不确定处按 Material Design 3 | 已确认 |
| 后端 | 第一版暂不需要远程后端，但可能需要本地元数据匹配模块 | 已初步确认 |
| 数据库 | 第一版优先 SQLite；不整体加密数据库；保留后续 SQLCipher 迁移可能 | 已初步确认 |
| 桌面端方案 | Tauri | 已确认 |
| 移动端方案 | 第一版不考虑 | 已推迟 |
| 部署方式 | 第一版自用，本地运行；稳定后再考虑安装包 | 已初步确认 |

| 测试工具 | 建议 Vitest + Playwright，待确认 | 建议待确认 |

## 3. 技术选择原则

### 3.1 优先原则

1. 优先简单。
2. 优先稳定。
3. 优先可维护。
4. 优先文档丰富。
5. 优先适合 AI Agent 修改。
6. 优先适合 MVP 快速验证。
7. 优先减少新手心智负担。

### 3.2 避免原则

1. 避免过早使用微服务。
2. 避免过早引入复杂云架构。
3. 避免过早引入复杂权限系统。
4. 避免为小项目引入大型框架组合。
5. 避免多个技术栈混用导致维护困难。
6. 避免没有明确理由的技术选择。

## 4. 软件形态选择

### 4.1 候选形态

| 形态 | 适合场景 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|---|
| Web 应用 | 需要跨平台访问、多人使用 | 易部署、跨平台 | 可能需要后端和部署，不符合第一版本地桌面偏好 | 暂不优先 |
| 桌面软件 | 本地工具、隐私数据、本地数据库 | 本地能力强，适合隐私记录 | 打包发布稍复杂 | 当前优先 |
| 移动 App | 手机高频使用场景 | 便携 | 开发和发布复杂 | 暂不优先，待确认 |
| 命令行工具 | 自动化、开发者工具 | 简单高效 | 不适合影评和资料浏览 | 暂不优先 |
| 浏览器插件 | 浏览器工作流增强 | 贴近网页场景 | API、权限、网站兼容限制较多 | 后续可评估 |

### 4.2 当前选择

第一版倾向 Windows 桌面软件。

### 4.3 选择理由

- 用户明确希望软件运行在 Windows 电脑端，并作为独立软件存在。
- 数据涉及私人成人内容记录，本地桌面形态更有利于隐私控制。
- 第一版不需要服务器同步，桌面本地应用能降低部署复杂度。

### 4.4 暂不选择的方案

- Web 应用：需要部署和更复杂的账号/数据安全设计。
- 移动 App：不符合当前首选平台。
- 浏览器插件：更贴近 MissAV 使用场景，但会引入浏览器权限、第三方页面结构变化和上架限制。

## 5. 前端技术

### 5.1 候选方案

| 技术 | 优点 | 缺点 | 适合程度 |
|---|---|---|---|
| 原生 HTML/CSS/JavaScript | 简单、无框架 | 复杂后维护困难 | 不建议作为第一选择 |
| React | 生态成熟、AI 示例多，适合复杂列表、搜索、表单和状态管理 | 对初学者有学习成本 | 当前建议 |
| Vue | 上手相对友好 | Agent 生成质量依项目而定 | 待评估 |
| Svelte | 简洁 | 生态相对小 | 待评估 |
| PySide6 | 适合 Python 桌面软件 | UI 美化成本较高 | 待评估 |
| Tkinter | Python 内置、简单 | 界面较老旧 | 待评估 |
| Tauri + Web 前端 | 桌面体验较好，体积相对小，权限和本地能力边界清晰 | 需要理解前端和 Rust/系统打包边界 | 当前建议 |
| Electron + Web 前端 | 桌面应用生态成熟，适合复杂 UI，Node 生态强 | 安装包和运行资源占用较大 | 备选 |

### 5.2 当前选择

已确认采用 React + TypeScript 作为界面层，配合 Tauri 作为桌面壳。

### 5.3 选择理由

理由：

1. Nvy 的界面包含影片列表、女优列表、复杂筛选、组合搜索、详情编辑、候选确认、轮播和设置页，已经超出简单表单工具范围。
2. React + TypeScript 对复杂状态和组件拆分更稳，AI Agent 生成、修改和测试的资料也更丰富。
3. Web UI 方便后续做更好的视觉设计；如果未来需要迁移到 Web 或在线版本，前端资产更容易复用。
4. Tauri 官方文档说明它可使用任意能编译到 HTML、JavaScript、CSS 的前端，并利用系统 WebView 来减小应用体积；这符合本项目第一版自用、本地运行、未来再打包的方向。
5. 用户已明确不接受传统资料管理系统风格，React + CSS 能更直接实现 Google / Material Design 3 暗色风格、轮播曲线、滚动渐隐、悬浮提示和现代下拉交互。

### 5.4 风险

风险：

1. React 和 TypeScript 对初学者有学习成本，需要通过小任务和清晰文档降低维护难度。
2. Tauri 引入 Rust 侧命令、权限和打包概念，虽然业务逻辑可尽量放在前端 / TypeScript 层，但文件、数据库、系统能力仍需要理解边界。
3. Windows WebView2 环境和 Tauri 打包细节需要在开发计划阶段验证。
4. UI 视觉要求较高，开发阶段需要把全局主题、组件状态、动效和响应式尺寸作为验收重点，避免回退成普通后台管理界面。

暂不优先选择 Electron 的原因：

1. Electron 官方文档说明它通过嵌入 Chromium 和 Node.js 来构建桌面应用，这让开发模型成熟、统一，但会带来更大的运行时和安装包体积。
2. Nvy 第一版是个人本地工具，不需要 Electron 级别的完整 Chromium 打包能力。
3. 如果 Tauri 在元数据请求、本地 SQLite、文件导入导出或 Windows 兼容性上遇到实质阻碍，Electron 仍保留为第一备选。

### 5.5 从 Tauri 迁移到 Electron 的可行性

结论：可以迁移，但迁移速度取决于第一版的工程边界设计。

迁移相对容易的部分：

1. React + TypeScript 前端页面、组件、样式和大部分交互状态可以复用。
2. 纯 TypeScript 的业务逻辑，例如搜索语法解析、标签归一化、年龄标签计算、显示名 fallback，可以复用。
3. SQLite 数据表设计、导入导出格式、图片缓存目录规则只要保持文档化和框架无关，也可以复用。

迁移需要重写或适配的部分：

1. Tauri 的 Rust command 需要改成 Electron main process / preload / IPC。
2. 文件系统、SQLite 访问、打开外部链接、选择导入导出文件等桌面能力需要重新接 Electron API。
3. Tauri 权限配置、打包配置、窗口配置不能直接迁移到 Electron。
4. 如果使用 Tauri 特有插件，需要替换为 Electron 或 Node 生态中的对应方案。

为了保留快速迁移能力，第一版应遵守以下约束：

1. 前端只调用统一的 `desktopApi` 接口，不直接散落调用 Tauri API。
2. 数据库操作通过仓储层 / 服务层封装，不让 UI 组件直接依赖具体 SQLite 驱动。
3. 搜索、标签、姓名归一化、年龄计算等核心规则优先写成纯 TypeScript 模块。
4. Tauri 侧只做系统能力适配，例如文件读写、数据库连接、窗口、路径和外部链接。
5. 不在 Rust 侧写大量业务规则，除非存在明确性能或安全理由。

判断：如果按上述边界开发，后期迁移到 Electron 属于中等成本，不是无痛一键迁移，但可以避免推倒重写。若第一版大量使用 Tauri/Rust 特有能力承载业务逻辑，则迁移成本会明显升高。

暂不优先选择 PySide6 的原因：

1. PySide6 适合 Python + Qt 桌面开发，但本项目需要较多卡片、列表、筛选、搜索、弹窗和后续可能的富交互视觉，Qt Widgets 的现代化界面成本更高。
2. Python 做元数据抓取和脚本处理方便，但桌面 UI 与打包分发对初学者并不一定更简单。
3. 如果后续决定优先用 Python 复用元数据处理生态，PySide6 可以重新评估。

资料依据：

- Tauri 官方文档：Tauri 用系统 WebView，应用不需要随包携带浏览器引擎，并支持任意 Web 前端。
- Electron 官方文档：Electron 通过嵌入 Chromium 和 Node.js，允许用 JavaScript、HTML、CSS 构建跨平台桌面应用。
- Qt for Python 官方文档：PySide6 是 Qt 的 Python 绑定，可用于 Python 桌面 GUI。

### 5.6 UI 样式实现方案

当前视觉方向已经确认：参考 Google Gemini / Google AI Studio 暗色模式，偏 Material Design 3，暗灰背景，淡暗紫和淡暗粉强调色。具体实现方案已确认采用 `Tailwind CSS + Radix UI primitives + CSS 变量`。

候选方案：

| 方案 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|
| 纯 CSS / CSS Modules | 依赖最少，完全可控 | 组件状态、下拉、tooltip、弹窗、可访问性和复杂动效需要大量手写 | 不推荐作为第一选择 |
| MUI | Material 生态成熟，表单、弹窗、下拉、主题能力完整 | 默认观感容易接近通用后台系统；要做 Gemini / AI Studio 风格需要大量主题覆盖；包体和抽象较重 | 备选 |
| Tailwind CSS | 样式迭代快，适合自定义暗色视觉和复杂间距/渐变 | 只负责样式，不提供下拉、弹窗、tooltip 等交互逻辑 | 推荐的一部分 |
| Radix UI primitives + Tailwind CSS + CSS 变量 | Radix 提供无样式但可访问的下拉、弹窗、tooltip 等交互基础；Tailwind 负责快速定制视觉；CSS 变量管理暗色主题、淡紫/淡粉和状态色 | 依赖比纯 CSS 多；需要建立自己的组件规范 | 已确认 |
| shadcn/ui | 基于 Radix 和 Tailwind，初始组件效率高 | 默认风格需要重做；组件复制进项目后需要维护 | 可参考，不建议照搬默认风格 |

已确认：第一版采用 `Tailwind CSS + Radix UI primitives + CSS 变量`。

推荐理由：

1. Nvy 的视觉不是标准后台管理界面，MUI 默认组件很容易偏“资料管理系统”，需要大量覆盖。
2. Tailwind 适合快速实现暗灰背景、淡暗紫/淡暗粉、渐变、圆角、hover、focus、滚动遮罩和响应式网格。
3. Radix 可以降低自写下拉栏、tooltip、弹窗、确认框、菜单时的可访问性和键盘交互风险。
4. CSS 变量可以集中管理主题色、边框、阴影、危险色和选中态，后续调视觉时不需要大面积改组件。
5. 该组合保留较强自定义能力，比较适合做接近 Gemini / AI Studio 的暗色风格。

暂不直接推荐 MUI 的原因：

1. 虽然 MUI 符合 Material Design 方向，但默认气质更像通用企业后台。
2. 用户已经明确不接受老式资料管理系统观感，MUI 若不重度定制容易走偏。
3. 第一版需要大量视觉微调和动效，轻量自定义组件更利于控制最终观感。

该方案已由用户确认。仍需遵守文档先行规则：在用户明确说“开始实现代码”前，不创建工程、不安装依赖。

## 6. 后端技术

### 6.1 是否需要后端

第一版暂不需要远程后端。后续在线同步版本可能需要后端。

需要根据以下问题判断：

- 是否需要账号系统？
- 是否需要多人共享数据？
- 是否需要云端同步？
- 是否需要调用第三方 API？
- 是否需要长期保存用户数据？
- 是否需要权限控制？
- 是否需要从不同设备访问？

### 6.2 候选方案

| 技术 | 优点 | 缺点 | 适合程度 |
|---|---|---|---|
| 不需要后端 | 最简单，适合第一版本地离线记录 | 无法实现云同步和跨设备 | 当前优先 |
| Node.js / Express | 生态成熟 | 需要学习后端概念 | 待评估 |
| Next.js API Routes | 前后端一体 | 需要理解框架约定 | 待评估 |
| Python / FastAPI | 清晰、文档好 | 需要部署后端 | 待评估 |
| Supabase | 快速获得数据库和认证 | 依赖第三方平台 | 待评估 |

### 6.3 当前选择

第一版不引入远程后端。仅保留未来在线同步扩展可能。

## 7. 数据存储

### 7.1 是否需要存储

需要。第一版必须长期保存本地账号、影片、女优、影评、关联关系和设置。

### 7.2 候选方案

| 存储方式 | 适合场景 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|---|
| 不存储 | 临时工具 | 最简单 | 无历史数据 | 待评估 |
| JSON 文件 | 小规模本地配置 | 简单可读 | 不适合复杂查询 | 待评估 |
| CSV 文件 | 表格类数据 | 易导出 | 结构能力弱 | 待评估 |
| SQLite | 本地结构化数据 | 稳定、轻量，适合账号隔离、影片、女优和关联关系 | 数据库文件默认不整体加密，需要配合密码哈希和导出加密 | 第一版优先 |
| SQLCipher | 需要整体加密数据库 | 基于 SQLite 生态，能提供数据库文件级加密 | 引入额外依赖、密钥管理和恢复复杂度 | 后续保留 |
| PostgreSQL | 云端复杂数据 | 强大可靠 | 对 MVP 可能过重 | 待评估 |
| Supabase | 快速云数据库 | 功能完整 | 依赖服务 | 待评估 |

### 7.3 当前选择

本地存储已确认；第一版优先使用 SQLite，不做数据库整体加密。密码必须哈希保存，导出文件支持可选加密。SQLCipher 或其他整体加密方案保留为后续升级方向。

### 7.4 API Key 加密保存候选方案

第一版已经确认 API Key 不能明文保存。候选方案如下：

| 方案 | 选择内容 | 选择理由 | 放弃或暂缓原因 | 当前判断 |
|---|---|---|---|---|
| Tauri Stronghold | 使用 Tauri 官方 Stronghold 插件保存 API Key | 与 Tauri 技术栈一致，面向敏感信息存储，适合第一版桌面应用 | 需要设计解锁方式、多账号隔离和迁移策略 | 已确认 |
| Windows 凭据管理器 / 系统 Keychain | 使用操作系统凭据存储能力 | Windows 原生，安全边界清晰 | 如果未来跨平台或导入导出迁移密钥，抽象成本更高 | 备选 |
| 账号密码派生密钥 + AES-GCM | 用本地账号密码派生密钥，加密 API Key 后存入 SQLite | 与账号绑定，导出迁移可控 | 自行承担更多密码学实现、错误处理和忘记密码风险 | 后续可评估 |
| 明文配置文件或明文 SQLite | 不加密保存 | 实现最简单 | 不符合已确认的安全要求 | 禁止 |

已确认：第一版采用 Tauri Stronghold 保存 API Key。SQLite 只保存接口类型、Base URL、模型型号 / 模型 ID、温度、最大输出长度、提示词等非敏感配置。API Key 不进入普通导出；需要迁移设备时，第一版可以要求用户重新填写 API Key。

影响：

1. 需要在架构中区分“普通账号设置”和“敏感密钥存储”。
2. 需要避免日志、错误提示、导出文件和调试信息泄露 API Key。
3. 若未来改用整体加密数据库，Stronghold 方案仍可保留，也可迁移到新的密钥管理策略。

## 8. AI / 第三方 API / 元数据来源

### 8.1 是否需要 AI 能力

用户希望第一版接入大模型，用户自行填写 API Key，通过首页下方文本框自然语言交互，检索本地数据库并推荐影片和女优。

当前决策：第一版实际调用大模型执行首页推荐和标题/简介自动翻译。翻译启用后，翻译结果保存到本地，避免重复翻译。

注意：影片标题和简介需要优先显示中文。第一版使用大模型翻译作为中文化方案，并保留用户手动编辑兜底。

第一版大模型设置建议采用供应商中立设计，字段至少包括：

- API Key，本地加密保存。
- 接口类型：OpenAI Responses API、OpenAI 兼容 Chat Completions、自定义接口。
- Base URL / API 地址。
- 模型型号 / 模型 ID，例如 `gpt-5.5`、`deepseek-v4-pro`。
- 供应商名称或备注。
- 温度、最大输出长度等常见参数。
- 每次推荐参考的女优和影片总数上限，默认 30，范围 0-999；0 表示不限数量。
- 翻译提示词。
- 首页推荐提示词。
- 首页推荐默认启用；配置缺失时不可用。
- 是否启用大模型翻译。

API Key 属于敏感信息，不能写入日志，不能提交到代码仓库，第一版需要本地加密保存。

OpenAI 官方 Responses API 当前通过 `POST /v1/responses` 创建模型响应，请求可包含 `model`、`input`、`instructions`、`max_output_tokens`、`temperature`、`stream`、`tools` 等字段。Nvy 的大模型设置需要支持该接口类型，同时保留 OpenAI 兼容 Chat Completions 和自定义接口，便于切换其他模型服务。模型型号 / 模型 ID 由用户自由填写，软件不预设或限制具体供应商型号。

### 8.2 是否需要第三方 API

第一版需要评估第三方元数据来源，用于实现“输入番号匹配影片封面和基础资料”“输入女优名匹配头像和女优资料”。

当前候选来源：

| 来源 | 可解决的问题 | 主要限制 | 当前判断 |
|---|---|---|---|
| gfriends/gfriends | 按女优名字或别名匹配头像图片 | 主要提供头像和别名匹配，不提供完整影片资料；图片来源和更新策略需要遵守项目说明 | 第一版女优头像优先来源，匹配后缓存图片到本地 |
| Yuukiy/JavSP | 多站点影片元数据抓取逻辑，可参考番号识别、封面、标题、女优等字段 | GPL-3.0 与附加限制需要评估；部分站点可能有 Cloudflare、Cookie、区域访问、登录或稳定性问题；不宜未经确认直接复制代码 | 第一版影片资料优先观察其多站点逻辑查询质量 |
| conanak99/jav-rest-api | 封装 DMM/FANZA 女优和影片 API，可返回女优三围、生日、头像、影片封面等字段 | 依赖 DMM API、网络、凭证和接口可用性；项目本身偏 API 服务封装，不是桌面端模块 | 女优资料来源；影片资料在 JavSP 兼容逻辑查不到时作为 fallback，并保留未来优先使用 DMM/FANZA API 的可能 |

第一版本地记录功能必须在无网络时可用；元数据匹配采用“自动匹配 + 候选确认 + 手动编辑兜底”。匹配失败或候选不满意时，用户必须能手动创建和自由编辑资料。

已确认的第一版匹配顺序：

1. 女优头像：优先查 gfriends。
2. 女优资料：查 DMM/FANZA API。
3. 影片资料：先参考 JavSP 的多站点逻辑观察查询质量。
4. 影片资料 fallback：JavSP 兼容逻辑查不到时，再查 DMM/FANZA API。
5. 后续保留把影片资料优先来源调整为 DMM/FANZA API 的可能。

新增技术影响：

- 女优姓名需要做归一化匹配：简体中文、曾用中文名、繁体中文、日文名、罗马音和大小写差异需要识别为同一女优。
- 影片和女优各自有独立标签库，需要支持标签增删改和引用检查。
- 影片搜索需要支持 `+` 组合条件、`#标签` 条件、作品类型筛选、排序叠加、年龄区间标签检索。
- 女优搜索需要支持模糊搜索、标签搜索、罩杯多选和显示名 fallback。
- 标签库需要支持中文规范名、繁简归一、同义词和关联词。第一版建议优先自建轻量词库，而不是直接引入复杂中文 NLP 服务。
- 中文标题和中文简介第一版使用大模型自动翻译，翻译结果缓存到本地；用户仍可手动编辑。

### 8.3 风险

可能风险：

- 网络不可用或第三方站点不可访问。
- 数据源接口变更、页面结构变更或反爬机制导致匹配失败。
- API 限流、Cookie、登录、区域访问和 Cloudflare 等限制。
- 开源项目许可证与附加条款限制直接复用代码。
- JavSP 只能作为逻辑参考，直接复用代码前需要单独确认许可证和附加条款。
- 影片资料来源优先级可能根据实际查询质量调整。
- 多语种姓名归一化和模糊搜索如果实现不严谨，可能导致同名误匹配。
- 中文标题和简介如果依赖外部翻译服务，会引入联网、隐私、成本和质量问题。
- 元数据可能不完整、过时或存在多候选误匹配。
- 图片外链可能失效；如果缓存本地，需要处理导入导出和存储空间。
- 已确认封面和头像缓存到本地，因此导入导出需要覆盖图片资源。
- 隐私问题：匹配请求会把番号或女优名发送给第三方来源。

## 9.3 包管理工具推荐

候选方案：

| 方案 | 优点 | 缺点 | 当前判断 |
|---|---|---|---|
| npm | Node 自带，学习成本最低，资料多 | 依赖安装速度和磁盘复用一般；大型前端项目依赖树更容易臃肿 | 备选 |
| pnpm | 安装快，磁盘占用低，依赖结构更严格，有利于发现隐藏依赖问题 | 需要额外安装；对完全没接触过前端的人多一个工具名 | Agent 推荐 |
| yarn | 成熟，生态可用 | 当前没有明显优于 pnpm 的理由 | 暂不推荐 |

已确认：第一版采用 `pnpm`。

推荐理由：

1. Tauri + React + TypeScript + Tailwind / Radix 这类前端依赖较多，pnpm 的安装速度和磁盘复用更合适。
2. pnpm 的依赖隔离更严格，可以更早暴露“代码依赖了未声明包”的问题。
3. pnpm 在现代前端项目中足够成熟，AI Agent 和社区资料都比较丰富。
4. 如果后续迁移到 Electron 或增加前端工具链，pnpm 仍然适用。

该方案已由用户确认。仍需遵守文档先行规则：在用户明确说“开始实现代码”前，不创建工程、不安装依赖。
