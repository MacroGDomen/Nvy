# 06 验收测试

## 1. 文档目的

本文档用于记录软件功能的验收标准。

验收测试的目标不是证明代码写得漂亮，而是证明软件行为符合用户需求。

每个核心功能都应该有明确的验收方式。

## 2. 验收原则

1. 验收标准必须具体。
2. 验收标准必须能被手动验证。
3. 验收标准必须和用户场景相关。
4. 不要只写“功能正常”。
5. 应该说明操作步骤、输入、期望结果。
6. 异常情况也需要验收。
7. MVP 功能必须有验收标准。

## 3. 测试环境

当前测试环境尚未最终确定，但第一版目标环境如下：

| 项目 | 内容 |
|---|---|
| 操作系统 | Windows 11 和 Windows 10 |
| 浏览器 | 不适用，除非后续采用 Web 技术或打开外部链接 |
| 设备 | Windows 电脑 |
| 软件版本 | 暂未确定 |
| 网络环境 | 本地核心功能应在无网络时可用；元数据匹配、大模型推荐和大模型翻译需要联网；大模型设置和提示词可离线保存 |
| 测试数据 | 本地测试账号、测试影片、测试女优、测试影评 |

## 4. 核心流程验收

### A001 本地记录核心流程验收

#### 对应功能

F001-F010、F012、F014-F038。

#### 前置条件

软件已安装或可运行，当前没有依赖服务器。

#### 操作步骤

1. 打开软件并选择离线使用。
2. 注册一个合法本地账号。
3. 使用该账号登录。
4. 新增一位女优并填写名字、三围、中文维基百科链接和个人评价。
5. 输入女优名并尝试匹配头像和基础资料。
6. 在候选结果中选择一个结果，或跳过匹配后手动录入。
7. 新增一部影片并输入番号。
8. 尝试匹配影片封面、标题和基础资料。
9. 在候选结果中选择一个结果，或跳过匹配后手动录入。
10. 将影片标记为单人作品。
11. 撰写影片影评。
12. 在影评中插入 `[01:23:45]` 格式时间戳。
13. 将单人作品关联到该女优。
14. 新增一部多人作品，导入演员名单，其中一位女优已收藏、两位未收藏；确认只关联已收藏女优。
15. 后续新增此前未收藏但出现在多人作品演员名单中的女优，确认软件弹出补关联确认，用户确认后多人作品补充关联。
16. 确认匹配得到的封面和头像已缓存到本地。
17. 导出当前账号数据，并分别测试默认普通导出和可选加密导出。
18. 关闭软件后重新打开并登录同一账号。
19. 查看影片详情、女优详情和首页轮播。
20. 在测试环境中导入此前导出的数据并检查恢复结果。
21. 删除影片或女优时验证二次确认；删除女优后确认影片仍保留。

#### 期望结果

软件能显示此前保存的账号、影片、影评、时间戳、女优资料、匹配资料、手动修改内容和关联关系；导入后数据能够恢复。

#### 通过标准

不需要手动保存，重启后数据仍存在且归属正确账号；导出文件可用于恢复资料库。

#### 状态

未开始。

## 7. 数据验收

需要重点验证：

- 账号数据是否正确保存。
- 密码是否不以明文方式保存。
- 影片、女优、影评和关联关系是否正确保存与读取。
- 影片作品类型是否正确限制必填项和女优关联行为。
- 多人作品是否只关联已收藏女优，并能在后续新增女优时弹出确认后补关联。
- 删除女优后影片是否仍保留。
- 影片标签库和女优标签库是否互相独立。
- 预置标签、标签同义词、繁简归一和关联标签是否能正确参与搜索。
- 年龄标签和 Cup 标签是否按规则自动生成。
- 元数据匹配得到的封面、头像和资料字段是否可保存与修改。
- 匹配得到的封面和头像是否缓存到本地，并随导入导出保持完整。
- 普通导出包是否包含 manifest、data、assets 和 checksums，且导入前会校验完整性。
- 加密导出是否不会暴露明文数据；普通导出和加密导出都不包含明文 API Key。
- 首页推荐是否按设置发送候选影片/女优元数据和用户文本全文。
- 推荐参考总数上限是否正确生效：默认值为 30；0 表示不限；1-999 表示影片和女优合计不超过该数量。
- 不同账号之间数据是否隔离。
- 自动保存是否可靠。
- 导入导出功能需要验证数据完整性、账号归属和版本兼容。

## 8. 界面验收

第一版界面需要验证：

- 用户能找到登录、注册、影片、女优、设置入口。
- 时间戳输入和添加按钮位置清楚。
- 影评编辑区域不会因为插入时间戳导致光标错乱。
- 错误提示清楚，不暴露底层异常。
- 首页对话框在第一版作为大模型推荐入口，配置有效时默认可用；配置缺失时应明确提示去设置页配置。
- 首页轮播展示所有已记录影片和女优，点击跳转正确。
- 影片详情页需要有 `重新翻译` 和 `取消翻译` 入口，并且不会导致原始标题和简介丢失。
- 第一版开发前需要完成首页、女优库页面、影片库页面、女优资料详情页、影片资料详情页和设置页面的草图确认；当前新版草图已确认通过。
- 第一版 UI 需要符合新版草图的 Google / Material Design 3 暗色风格，不应呈现传统资料管理系统观感。
- 左侧状态栏、顶部搜索栏、首页轮播、帘式滚动、输入框、按钮和下拉栏需要按 `docs/08_page_wireframes.md` 的参考文件和交互要求验收。

## 10. 安全与隐私验收

需要重点验证：

- 成人内容相关私人记录是否只保存在预期本地位置。
- 密码是否不会明文保存。
- 第一版数据库文件不做整体加密，该限制需要在安全说明中明确。
- 加密导出文件是否必须输入正确密码才能恢复。
- API Key 是否不会出现在日志或代码仓库中。
- 大模型提示词和 API 设置是否按账号保存并隔离。
- 未登录状态下不能访问账号数据。
- 账号 A 不能访问账号 B 的影片、女优、影评和设置。
- 元数据匹配请求会发送番号或女优名，需要在产品说明中明确。
- 调用大模型 API 前，需要明确用户输入和本地资料中哪些内容会被发送；推荐会发送用户输入、候选影片/女优元数据和用户文本全文，翻译会发送标题和简介原文。
- 推荐发送内容不得包含密码、密码哈希、API Key 或本地绝对路径。
- API Key 使用 Tauri Stronghold 本地加密保存；验收时以“本地存储、日志、导出文件均无明文 API Key”为通过标准。
