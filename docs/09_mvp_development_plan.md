# 09 第一版 MVP 开发计划

## 1. 文档目的

本文档用于把 Nvy 第一版 MVP 拆分成后续可以逐步实现的小任务。

当前仍处于开发计划阶段，不进入代码实现阶段。本计划只说明后续实现时建议创建或修改的文件 / 模块，不实际创建工程、不安装依赖、不写业务代码。

## 2. 依据文档

本计划基于以下文档整理：

1. `AGENTS.md`
2. `docs/02_product_spec.md`
3. `docs/05_feature_backlog.md`
4. `docs/06_acceptance_tests.md`
5. `docs/03_tech_stack.md`
6. `docs/04_architecture_notes.md`
7. `docs/08_page_wireframes.md`
8. `docs/07_decision_log.md`
9. `docs/00_project_brief.md`
10. `README.md`

其中 `docs/02_product_spec.md` 是最高优先级需求来源。

## 3. 当前结论

### 3.1 未发现阻塞开发计划的未确认事项

当前 MVP 范围、技术栈、UI 方向、运行平台、数据存储、大模型接入、导入导出格式、图片缓存和页面草图均已确认，可以进入开发计划拆分。

### 3.2 需要指出的文档状态冲突

- `AGENTS.md` 第 12 节仍写着“当前项目状态：需求访谈与文档初始化阶段”。
- `docs/07_decision_log.md` 的 D021、`docs/00_project_brief.md`、`README.md` 和用户本轮说明均确认“需求访谈阶段结束，进入第一版 MVP 开发计划阶段”。

处理方式：本计划按用户本轮说明和 D021 执行，只做开发计划，不进入代码实现。

### 3.3 第一版 MVP 不扩大范围

本计划只覆盖已确认为 P0 的第一版能力。以下内容不进入第一版实现计划：

- 在线服务器同步。
- 云端账号体系。
- 多设备同步。
- 成人影片播放、下载、托管或分发。
- 浏览器插件。
- 安装包发布、自动更新。
- 复杂大模型长期记忆、多轮策略和自动调参。
- 全局撤销 / 重做。
- 独立草稿系统。
- 数据库整体加密。

## 4. 推荐总体顺序

推荐顺序遵循“先跑通本地可运行骨架和数据闭环，再做资料管理，再做搜索标签，再做外部能力和视觉细节”的原则。

1. 工程骨架与基础约束。
2. 本地数据模型、数据库和账号隔离。
3. 登录注册与应用外壳。
4. 影片 / 女优基础 CRUD 和自动保存。
5. 作品类型、关联关系、删除确认和时间戳。
6. 标签库、姓名归一化、组合搜索和自动标签。
7. 图片缓存和导入导出。
8. 元数据匹配和候选确认。
9. 大模型设置、Stronghold、推荐和翻译。
10. 首页轮播、库页面、详情页、设置页视觉完善。
11. 1999 条规模、离线、隐私和回归验收。

## 5. 开发阶段划分

## 阶段 0：工程骨架与开发基线

目标：建立最小可运行的 Tauri + React + TypeScript + SQLite 项目基线，为后续小任务提供稳定落点。

必须先做。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 0.1 初始化 Tauri + React + TypeScript 工程 | `package.json`、`pnpm-lock.yaml`、`src/`、`src-tauri/`、`tsconfig.json`、`vite.config.ts` | 使用 `pnpm` 能启动开发环境；窗口可打开空白 Nvy 应用；不包含业务功能 |
| 0.2 接入 Tailwind、Radix primitives 和 CSS 变量主题骨架 | `tailwind.config.*`、`src/styles/`、`src/components/ui/` | 暗色背景、基础文本色、强调色、按钮、输入框、弹窗、tooltip 和下拉基础样式可复用 |
| 0.3 建立目录边界 | `src/app/`、`src/pages/`、`src/components/`、`src/domain/`、`src/services/`、`src/lib/`、`src-tauri/src/` | UI、业务规则、数据访问、桌面 API 适配边界清晰；页面组件不直接操作 SQLite |
| 0.4 建立统一 `desktopApi` 适配层 | `src/services/desktopApi/`、`src-tauri/src/commands/` | 前端只通过统一接口调用 Tauri 能力；后续迁移 Electron 时不需要大面积改页面 |
| 0.5 建立最小测试基线 | `vitest` 配置、可选 `playwright` 配置、`tests/` 或 `src/**/*.test.ts` | 至少能运行一个纯逻辑单元测试；后续规则模块可持续加测试 |

实现阶段风险：

- Tauri、SQLite、Stronghold、文件系统权限需要尽早验证。
- Windows 10 / 11 的 WebView2 环境需要在后续验收中确认。
- 不要在此阶段加入未确认依赖。

第一轮真正开始写代码时，最小第一个任务应是：任务 0.1，只创建最小 Tauri + React + TypeScript + pnpm 工程骨架，并确认开发窗口能启动。

## 阶段 1：数据模型、数据库和本地账号

目标：先把本地账号、SQLite schema、账号隔离和基础仓储层做好，让后续页面都有可靠的数据落点。

必须先做。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 1.1 设计 SQLite schema 与迁移机制 | `src-tauri/src/db/`、`migrations/`、`src/domain/schema/` | 覆盖账号、影片、女优、关联、标签、设置、翻译状态；数据库可初始化和重复启动 |
| 1.2 实现本地账号表和密码哈希 | `auth` 模块、`accounts` repository | 合法用户名 / 密码可注册；密码不明文保存；非法输入被拒绝 |
| 1.3 实现登录和当前账号上下文 | `authService`、`sessionStore` | 登录后能获得当前账号 id；未登录不能读取账号数据 |
| 1.4 实现账号数据隔离基础查询 | 所有 repository 的 `account_id` 约束 | 账号 A 创建的数据账号 B 不可见 |
| 1.5 初始化默认账号设置和默认提示词 | `settings` repository、默认值模块 | 新账号拥有默认推荐参考总数 30、默认推荐提示词、默认翻译提示词 |
| 1.6 初始化预置标签库 | `tagRepository`、`tagService` | 新账号有影片预置标签和女优预置标签；重复初始化不产生重复项 |

验收映射：A002、A003、A030、A032、A033，异常 E001-E006。

实现阶段风险：

- 密码哈希算法需要选择成熟方案，不能自制哈希。
- 所有业务表都需要账号隔离，不能只在 UI 层过滤。
- schema 版本要从第一版开始记录，方便导入导出和后续迁移。

## 阶段 2：应用外壳、登录注册和基础导航

目标：建立用户可以进入应用的最小界面，并打通登录后的全局布局。

必须先做。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 2.1 登录 / 注册页 | `src/pages/AuthPage.tsx`、`auth` 组件 | 用户能注册、登录；错误提示清楚，不暴露底层错误 |
| 2.2 登录态路由保护 | `src/app/router`、`sessionStore` | 未登录只能看到登录页；登录后进入首页 |
| 2.3 全局左侧折叠状态栏 | `src/components/layout/Sidebar.tsx` | 默认折叠只显示图标；hover 有 tooltip；点击 logo 可展开 |
| 2.4 基础页面路由 | `HomePage`、`VideosPage`、`ActressesPage`、`SettingsPage` | 首页、影片库、女优库、设置页可切换 |
| 2.5 全局错误提示与确认弹窗基础 | `Toast`、`ConfirmDialog` | 错误提示可复用；二次确认可复用 |

验收映射：A001、A002、A003、A043，异常 E001-E006。

实现阶段风险：

- UI 不能退回传统后台系统观感。
- 这一阶段只做页面入口和外壳，不做复杂业务页面。

## 阶段 3：影片和女优基础资料管理

目标：先做手动录入、编辑、查看和自动保存，不依赖元数据匹配和大模型。

必须先做。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 3.1 影片数据 repository 和 service | `videoRepository`、`videoService` | 可创建、读取、更新影片基础字段 |
| 3.2 女优数据 repository 和 service | `actressRepository`、`actressService` | 可创建、读取、更新女优基础字段 |
| 3.3 影片列表最小页面 | `VideosPage`、`VideoCard` | 能展示当前账号影片；点击进入详情 |
| 3.4 女优列表最小页面 | `ActressesPage`、`ActressCard` | 能展示当前账号女优；点击进入详情 |
| 3.5 影片详情基础编辑 | `VideoDetailPage`、影片表单组件 | 番号、标题、简介、发行日期、片长、来源链接、作品类型、影评可编辑 |
| 3.6 女优详情基础编辑 | `ActressDetailPage`、女优表单组件 | 名字、多语种姓名、三围、生日、身高、出道日期、维基链接、评价可编辑 |
| 3.7 自动保存机制 | `autosave` hook / service | 编辑后无需手动保存；重启后数据仍存在 |
| 3.8 维基百科链接打开 | `desktopApi.openExternal` | 女优详情中的中文维基百科链接可打开；无链接时不报错 |

验收映射：A004、A006、A007、A011、A017、A018。

实现阶段风险：

- 自动保存要避免每次按键都阻塞输入。
- 第一版不做草稿，保存失败时要保留当前编辑内容。
- 日期和数字字段需要统一格式，避免后续搜索和自动标签困难。

## 阶段 4：作品类型、关联关系、时间戳和删除

目标：完成本地记录核心闭环里最关键的业务规则。

必须先做。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 4.1 影片与女优关联表和服务 | `videoActressRepository`、`associationService` | 影片可关联一个或多个已存在女优；女优详情可查看相关影片 |
| 4.2 作品类型必填规则 | `videoValidation`、详情页保存逻辑 | 番号和作品类型必填；单人作品必须关联女优；多人和素人不强制 |
| 4.3 素人手动关联 | 影片详情关联组件 | 素人作品允许手动关联已收藏女优 |
| 4.4 多人作品演员姓名文本保存 | `video.actor_names` 相关字段 / 表 | 多人作品可保存演员名单文本；未收藏女优不自动创建 |
| 4.5 多人作品已有女优自动关联 | `associationService`、`nameMatchService` | 导入演员名单时只自动关联已收藏且匹配的女优 |
| 4.6 后续新增女优补关联确认 | `associationSuggestionService`、确认弹窗 | 新增女优匹配既有多人作品演员名单时弹出确认；取消则不关联 |
| 4.7 影评时间戳插入 | `TimestampInput`、影评编辑器 | 有效时分秒在当前光标插入 `[01:23:45]`；无效输入不污染影评 |
| 4.8 删除二次确认 | 删除按钮、`ConfirmDialog`、repository 删除方法 | 删除影片 / 女优前必须确认；删除女优不删除影片 |

验收映射：A004、A005、A019、A020、A021，异常 E007、E012-E014、E017。

实现阶段风险：

- 多人作品补关联依赖姓名匹配质量，必须保留用户确认。
- 删除是永久删除，确认文案要清楚。
- 时间戳插入要维护光标位置，不能把文本插到末尾。

## 阶段 5：标签库、姓名归一化、搜索和自动标签

目标：让资料库真正可检索，并覆盖第一版已确认的标签和多语种姓名规则。

可以在阶段 3-4 之后做；必须在大模型推荐前完成基础候选检索。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 5.1 标签库 CRUD | `tagRepository`、`tagService`、设置页标签分区 | 影片标签库和女优标签库独立；可新增、修改、删除 |
| 5.2 标签校验 | `tagValidation` | 标签原则上必须中文；`#NTR` 作为例外；未成年人相关标签被拒绝或提示 |
| 5.3 标签同义词和关联词 | `tagRelationService` | 搜索同义词 / 繁体 / 关联词能命中规范标签记录 |
| 5.4 女优姓名归一化规则 | `nameNormalize.ts` | 繁简、大小写、全角半角、空格和常见分隔符归一化 |
| 5.5 女优多语种搜索 | `actressSearchService` | 简中、繁中、日文、罗马音、曾用名和别名均参与搜索 |
| 5.6 女优显示名 fallback | `displayNameService` | 显示名类型和单个女优默认显示类型生效；缺失时按文档顺序回退 |
| 5.7 影片组合搜索解析 | `videoSearchParser` | 支持关键词、`#标签`、`+` 条件叠加、作品类型筛选、排序 |
| 5.8 年龄标签解析和检索 | `ageTagService`、搜索服务 | 支持 `#[22-33]`、`#[22-]`、`#[25]`、`#[-25]` |
| 5.9 自动年龄标签 | `autoTagService` | 单人作品且生日 / 发行日期存在时自动添加年龄标签；缺失任一日期不添加 |
| 5.10 自动 Cup 标签 | `autoTagService` | 单人作品且女优罩杯存在时自动添加 `#xxCup` |
| 5.11 女优罩杯多选筛选 | 女优库筛选组件、搜索服务 | A-K 且排除 I/J；支持全选和多选 |

验收映射：A022-A028、A030、A031，异常 E015、E016、E018。

实现阶段风险：

- 搜索规则复杂，建议纯 TypeScript 模块优先写单元测试。
- 标签同义词 / 关联词由用户维护，不能把未确认的 Agent 推荐默认写成正式标签。
- 自动标签要允许用户后续删除或编辑。

## 阶段 6：图片缓存和数据导入导出

目标：实现备份、迁移、恢复和离线图片显示能力。

可以在基础 CRUD 后做；元数据匹配前可先实现本地图片占位和缓存目录。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 6.1 应用数据目录适配 | `desktopApi.path`、`src-tauri` 文件命令 | 使用应用专属数据目录；数据库、缓存、导出路径清晰 |
| 6.2 图片缓存服务 | `imageCacheService`、`cache/covers`、`cache/actresses` | 封面和头像以内部 id 命名；数据库保存相对路径 |
| 6.3 本地图片选择 / 替换 | 图片上传组件、缓存服务 | 用户可手动设置封面 / 头像；重启后仍显示 |
| 6.4 普通导出 `.nvyzip` | `exportService`、压缩 / manifest / checksums 模块 | 导出包含 manifest、data、assets、checksums；不含明文 API Key |
| 6.5 加密导出 `.nvyenc` | `encryptedExportService` | 加密导出需要密码；导出内容不可直接明文读取 |
| 6.6 导入校验 | `importValidator` | manifest 或 checksums 缺失 / 损坏时拒绝导入，不破坏现有数据 |
| 6.7 普通导入恢复 | `importService` | 导入后账号、影片、女优、影评、关联、标签和图片引用保持完整 |
| 6.8 加密导入恢复 | `encryptedImportService` | 密码正确可导入，密码错误拒绝且不破坏现有数据 |

验收映射：A008、A014、A015、A040，异常 E010、E011、E022。

实现阶段风险：

- 普通导出是明文 zip，本身会暴露隐私，需要界面提示。
- 导入失败必须事务化或先写临时区，不能破坏当前数据库。
- 导出不能包含明文 API Key，也不能包含本地绝对路径。

## 阶段 7：元数据匹配和候选确认

目标：降低录入成本，但必须保证匹配失败或误匹配时可以手动兜底。

可后做；不应阻塞手动资料管理闭环。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 7.1 元数据匹配统一接口 | `metadataProvider`、`metadataService` | 影片和女优匹配都返回统一候选结构 |
| 7.2 女优头像来源适配 | `gfriendsProvider` 或等价适配器 | 输入女优名可尝试匹配头像候选；失败允许手动编辑 |
| 7.3 女优资料来源适配 | `dmmFanzaActressProvider` | 输入女优名可尝试匹配三围、生日、头像等候选资料 |
| 7.4 影片资料来源适配 | `javspCompatibleProvider`、`dmmFanzaVideoProvider` | 输入番号可尝试匹配封面、标题、女优、发行日期等候选资料 |
| 7.5 候选确认弹窗 | `MetadataCandidateDialog` | 多候选时用户可选择、跳过或手动编辑；不静默覆盖 |
| 7.6 匹配结果写入和图片缓存 | `metadataApplyService`、`imageCacheService` | 确认候选后资料写入当前账号，封面 / 头像缓存本地 |
| 7.7 匹配失败兜底 | 详情页匹配按钮和错误处理 | 网络失败、无结果、接口失败时提示并允许继续手动录入 |

验收映射：A009、A010、A013、A015，异常 E009。

实现阶段风险：

- JavSP、gfriends、DMM/FANZA API 的许可证、接口稳定性、访问限制和地区限制需要开发时验证。
- 不应直接复制许可证边界未确认的代码。
- 元数据请求会发送番号或女优名，需要在设置或匹配流程中有明确说明。

## 阶段 8：大模型设置、Stronghold、推荐和翻译

目标：实现第一版实际调用大模型的 P0 能力，同时控制 API Key、隐私发送范围和失败兜底。

必须在基础资料、搜索和设置页具备后做。

| 任务 | 后续可能涉及文件 / 模块 | 验收标准 |
|---|---|---|
| 8.1 大模型设置 UI | `SettingsPage`、`LlmSettingsSection` | 可编辑接口类型、Base URL、模型型号 / 模型 ID、供应商备注、温度、最大输出长度、参考总数上限 |
| 8.2 推荐参考总数校验 | `settingsValidation` | 默认 30；只接受 0-999；非数字、负数、大于 999 被拒绝 |
| 8.3 提示词管理 | `PromptSettingsSection`、默认提示词模块 | 翻译提示词和首页推荐提示词可编辑、保存、重启后恢复 |
| 8.4 API Key Stronghold 存储 | `secretService`、Tauri Stronghold 命令 | 本地存储、日志、导出文件中不出现明文 API Key |
| 8.5 大模型客户端抽象 | `llmClient`、`responsesClient`、`chatCompletionsClient`、`customClient` | 支持 OpenAI Responses API、OpenAI 兼容 Chat Completions、自定义接口类型 |
| 8.6 首页推荐候选构造 | `recommendationCandidateService` | 先本地检索候选，再按参考总数上限截取；0 表示不限 |
| 8.7 首页推荐调用 | `HomeRecommendationBox`、`recommendationService` | 配置有效时输入偏好可调用模型并显示推荐影片、女优和理由 |
| 8.8 推荐发送边界保护 | `recommendationPayloadBuilder` | 发送用户输入、候选元数据和用户文本全文；不发送密码、API Key、本地绝对路径 |
| 8.9 自动翻译触发 | `translationService` | 启用翻译后，原始标题 / 简介存在且中文字段为空时自动翻译并缓存 |
| 8.10 重新翻译 | 影片详情翻译按钮、`translationService` | 点击后重新调用模型并覆盖中文标题 / 简介，保留原始字段 |
| 8.11 取消翻译 | 影片详情翻译按钮、状态处理 | 清空当前中文翻译结果，回到原文显示；请求返回后不得写回已取消结果 |
| 8.12 大模型失败处理 | `llmErrorHandler` | API Key 缺失、网络失败、接口失败时提示清楚，不破坏现有资料 |

验收映射：A012、A029、A032-A039、A042，异常 E019-E021、E023。

实现阶段风险：

- Stronghold 的解锁方式、多账号隔离和迁移策略需要开发时验证。
- 首页推荐发送用户文本全文，隐私提示必须明确。
- 上限为 0 且候选很多时可能超出模型上下文或成本过高，需要开发时处理失败。
- OpenAI Responses API 和兼容接口参数不完全一致，需要隔离客户端实现。

## 6. 必须先做与可以后做

### 6.1 必须先做

1. 阶段 0：工程骨架与开发基线。
2. 阶段 1：数据模型、数据库和本地账号。
3. 阶段 2：应用外壳、登录注册和基础导航。
4. 阶段 3：影片和女优基础资料管理。
5. 阶段 4：作品类型、关联关系、时间戳和删除。

原因：这些任务构成 Nvy 的本地记录核心闭环。没有它们，后续元数据匹配、大模型推荐、导入导出和视觉完善都没有稳定基础。

### 6.2 可以后做

1. 元数据匹配和候选确认：不应阻塞手动录入闭环。
2. 大模型推荐和自动翻译：需要依赖设置、搜索、资料数据和 Stronghold。
3. 首页轮播高级动效：可先显示基础数据，再优化曲线、缩放和自动轮播。
4. 1999 条规模性能优化：开发中持续关注，最终阶段集中验收。
5. 视觉细节精修：全局主题和基础组件应早做，页面动效和细节可后续迭代。

## 7. 暂不进入第一版

以下任务不安排到第一版 MVP 开发任务中：

| 功能 | 原因 |
|---|---|
| 在线服务器同步 | 第一版先验证本地记录闭环，云同步会显著增加账号、安全和冲突处理复杂度 |
| 云端账号体系 | 第一版使用本地账号即可满足账号隔离 |
| 成人影片播放 / 下载 | 软件定位是记录工具，不承担内容播放、下载或分发 |
| 浏览器插件 | 会引入浏览器权限和第三方网站结构变化风险 |
| 安装包发布 / 自动更新 | 第一版先自用，本地运行验证稳定后再考虑 |
| 数据库整体加密 | 第一版不整体加密数据库，采用密码哈希、Stronghold 和可选加密导出 |
| 撤销 / 重做 | 增加编辑状态复杂度，第一版不做 |
| 独立草稿系统 | 第一版使用自动保存，不做额外草稿状态 |
| 大模型复杂长期记忆和多轮策略 | 第一版只做基于本地候选资料的推荐和标题 / 简介翻译 |

## 8. 开发时必须验证的风险

1. Tauri + SQLite 在 Windows 10 / 11 下的本地数据库读写、迁移、事务和文件权限。
2. Tauri Stronghold 保存 API Key 的解锁方式、多账号隔离、重启恢复和不导出明文。
3. 密码哈希方案是否成熟、安全，且不会把密码明文写入日志。
4. 元数据来源的许可证、接口稳定性、地区访问、Cloudflare / 登录 / Cookie / 限流风险。
5. 直接或间接使用 JavSP 逻辑时的许可证边界，未经确认不要复制代码。
6. 图片缓存路径、文件命名和导入导出资源恢复是否稳定。
7. 导入失败、导入包损坏、加密密码错误时是否完全不破坏现有数据库。
8. 首页推荐 payload 是否严格排除密码、API Key、本地绝对路径和非候选资料。
9. 推荐参考总数上限为 0 时是否可能造成成本、上下文长度或请求失败问题。
10. 自动翻译取消后，晚返回的请求不能覆盖已取消状态。
11. 标签同义 / 关联检索和女优姓名归一化是否存在误匹配，需要候选确认兜底。
12. 1999 条记录下列表滚动、搜索、首页轮播、导入导出是否仍可用。
13. UI 是否符合已确认的 Google Gemini / Google AI Studio 暗色方向，而不是传统后台管理系统。

## 9. 第一轮代码实现建议

当用户明确说“开始实现代码”后，第一轮最小任务建议是：

> 创建最小 Tauri + React + TypeScript + pnpm 工程骨架，只实现一个可启动的 Nvy 空壳窗口和基础目录结构，不接数据库、不做业务页面、不安装超出已确认技术栈的依赖。

验收标准：

1. 使用 `pnpm` 可安装并启动开发环境。
2. Windows 上能打开 Nvy 应用窗口。
3. 页面只显示最小应用壳或占位内容。
4. 项目目录为后续阶段预留 `src`、`src-tauri`、UI、domain、services 等边界。
5. 不包含影片、女优、账号、大模型、导入导出等业务实现。

这样可以把第一步控制在最小范围内，先验证工具链和运行环境，再进入数据库和账号实现。
