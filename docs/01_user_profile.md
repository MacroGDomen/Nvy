# 01 用户画像

## 1. 文档目的

本文档用于记录本软件的目标用户画像，以及项目创建者的背景、能力边界和技术偏好。

注意：

- “目标用户”是最终使用这个软件的人。
- “项目创建者”是提出软件想法并使用 Coding Agent 开发软件的人。
- 两者可能是同一个人，也可能不是同一个人。

## 2. 目标用户画像

### 2.1 核心目标用户

核心目标用户是项目创建者本人，以及具有类似需求的成年个人用户。

用户在 Windows 电脑上通过第三方成人网站浏览和观看成人影片，但希望用独立本地软件保存自己的私人影评、影片时间点记录、女优评价和收藏资料。

### 2.2 用户基本特征

| 项目 | 内容 |
|---|---|
| 用户类型 | 成年个人用户 |
| 年龄范围 | 必须为成年人，具体年龄范围待确认 |
| 专业背景 | 不限定 |
| 技术水平 | 能够安装和使用 Windows 桌面软件；是否能配置 API Key 待确认 |
| 使用设备 | Windows 电脑 |
| 使用频率 | 与成人影片观看频率相关，待确认 |
| 使用环境 | 私人电脑、本地使用为主 |

## 3. 用户痛点

目标用户当前遇到的问题：

| 编号 | 痛点 | 现有解决方式 | 现有方式的问题 |
|---|---|---|---|
| P001 | 观看成人影片后，具体观后感和时间点感受容易遗忘 | 依赖网站收藏夹或记忆 | 收藏夹只保存影片，不保存个人影评和具体时间点 |
| P002 | 对感兴趣女优缺少个人化评价和关联影片整理 | 在网站内继续搜索或凭记忆回看 | 信息分散，难以形成自己的资料库 |
| P003 | 想记录某部影片何时观看、何时撰写影评 | 网站通常不提供私人观影记录 | 之后查看收藏时无法回忆当时感受 |
| P004 | 希望仅输入番号或女优名就自动补全封面、头像和基础资料 | 手动查找图片和资料 | 录入成本高，资料格式不统一 |
| P005 | 希望根据偏好描述寻找自己记录过或可能感兴趣的影片/女优 | 手动翻收藏或搜索网站 | 查找效率低，无法基于个人记录推荐；该能力已推迟到后续版本 |

## 4. 用户目标

用户希望通过本软件达成：

- 建立本地成人影片私人资料库。
- 为影片撰写影评和时间点记录。
- 为女优撰写个人评价，并查看其相关影片。
- 登录不同本地账号时看到彼此隔离的数据。
- 输入番号或女优名后尽量自动匹配封面、头像和基础资料，并允许自由修改。
- 匹配成功后将封面和头像缓存到本地。
- 导入导出本地数据，用于备份、迁移和恢复；默认普通导出，加密导出可选。
- 在首页轮播中看到所有已记录影片和女优。
- 以 `[01:23:45]` 格式记录影片时间点。
- 对单人作品建立女优关联；多人作品不添加或关联女优。
- 删除影片或女优前进行二次确认，避免误删。
- 后续可以通过自然语言描述，让大模型帮助检索或推荐记录中的影片和女优。

## 5. 用户使用场景

### 场景 1：记录影片影评

- 触发条件：用户在第三方成人网站观看或浏览到一部想记录的影片。
- 用户目标：保存影片番号、封面、名称和个人影评。
- 操作过程：用户打开本软件，登录本地账号，新增影片，填写影片信息并撰写影评。
- 期望结果：软件自动保存记录，后续用户可以按影片查看。

### 场景 2：记录影片时间点

- 触发条件：用户希望记录影片中某个值得回看的时间点。
- 用户目标：在影评中快速插入格式统一的时间戳。
- 操作过程：用户在时间输入框填写时、分、秒，点击添加，软件在当前光标处插入时间戳。
- 期望结果：影评文本中出现可读时间点，用户后续能快速定位记忆。

### 场景 3：整理女优资料

- 触发条件：用户对某位女优感兴趣，希望长期记录个人评价。
- 用户目标：保存女优资料、个人评价和相关影片。
- 操作过程：用户新增女优，填写名字、头像、三围、中文维基百科链接和评价，并将影片关联到该女优。
- 期望结果：用户进入女优详情页时，可以看到其资料、个人评价和已记录影片。

### 场景 4：通过描述寻找内容

- 触发条件：用户不知道今天想看哪部影片或哪位女优。
- 用户目标：通过自然语言描述偏好，让软件推荐可能匹配的影片或女优。
- 操作过程：用户在首页对话框输入偏好描述，软件调用用户自行配置的大模型 API，并基于本地资料库返回结果。
- 期望结果：用户获得候选影片或女优列表。
- 当前状态：不进入第一版 MVP；第一版只保留首页对话框样式。

### 场景 5：自动匹配基础资料

- 触发条件：用户新增影片或女优时，希望减少手动查找资料。
- 用户目标：输入番号或女优名后自动匹配封面、头像、标题、三围、生日等基础资料。
- 操作过程：用户输入番号或女优名，软件尝试从候选元数据来源匹配资料；用户检查结果并自由修改。
- 期望结果：匹配成功时减少录入成本；匹配失败时仍可手动创建记录。

## 9. 对技术选型的影响

由于项目创建者不熟悉传统软件开发，因此技术选型应遵循：

1. 优先简单。
2. 优先文档丰富。
3. 优先生态成熟。
4. 优先容易让 Coding Agent 稳定生成代码。
5. 避免过早引入复杂工程化。
6. 避免微服务、复杂云架构和过度抽象。
7. 每个技术选择都需要解释原因、替代方案和风险。
