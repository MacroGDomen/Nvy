# 04 架构说明

## 1. 文档目的

本文档用于记录项目的架构设计、模块划分、数据流、外部依赖和未来扩展方向。

当前项目仍处于需求访谈阶段，因此本文档只记录初步架构假设，不作为最终实现依据。

## 2. 当前架构状态

当前状态：第一版采用 Tauri + React + TypeScript + SQLite 的本地单体桌面应用。

原因：

- 软件形态已初步确认为 Windows 独立桌面软件。
- 技术栈已确认：Tauri + React + TypeScript + SQLite。
- 核心功能已明确为本地账号、影片记录、女优记录、影评、时间戳、自动保存、数据导入导出、影片元数据匹配和女优元数据匹配。
- 数据存储已确认需要本地持久化；第一版优先 SQLite，不整体加密数据库，使用密码哈希和可选加密导出；匹配到的封面和头像缓存到本地。
- 第一版暂不实现服务器同步；大模型设置、模型参数、提示词管理、首页推荐调用、标题/简介自动翻译和 API Key 本地加密保存进入 MVP。
- 第一版 UI 方向改为 Google Gemini / Google AI Studio 暗色风格，优先使用 Material Design 3 的交互和组件思想；具体页面草图以 `docs/08_page_wireframes.md` 为准。

## 3. 架构设计原则

1. 先满足 MVP，不做过度设计。
2. 模块职责清晰。
3. 数据流尽量简单。
4. 优先让项目创建者能理解。
5. 优先让 Coding Agent 能稳定维护。
6. 避免隐藏复杂度。
7. 避免为了未来可能性牺牲当前简单性。

## 4. 可能的架构形态

### 4.1 本地单体应用

适合：

- 个人工具。
- 需要本地账号。
- 不需要多人协作。
- 数据保存在本地。
- 需要访问本地文件、图片资源或本地数据库。

可能结构：

```text
应用界面
  ↓
业务逻辑
  ↓
本地账号 / 本地数据库 / 本地图片文件 / 设置
```

优点：

- 简单。
- 易启动。
- 不需要服务器。
- 适合 MVP。

缺点：

- 多设备同步困难。
- 多人协作困难。
- 发布和升级需要额外考虑。

当前判断：第一版优先采用该架构方向。

### 4.2 Web 前后端一体应用

适合：

- 需要浏览器访问。
- 未来可能部署到云端。
- 需要账号系统。
- 需要数据库。
- 需要跨设备访问。

可能结构：

```
浏览器界面
  ↓
应用服务
  ↓
数据库 / 文件存储 / 第三方 API
```

优点：

- 跨平台。
- 易分享。
- 适合多人访问。

缺点：

- 需要部署。
- 需要考虑后端。
- 需要考虑账号、安全、数据库。

当前判断：后续在线同步版本可评估，第一版暂不采用。

### 4.3 前后端分离架构

适合：

- 中大型项目。
- 多客户端。
- 复杂 API。
- 明确需要扩展。

可能结构：

```
前端应用
  ↓
后端 API
  ↓
数据库 / 第三方服务
```

优点：

- 边界清晰。
- 可扩展性强。

缺点：

- 对 MVP 可能过度复杂。
- 对初学者学习成本较高。
- 需要维护更多工程。

当前判断：暂不优先，除非需求证明必要。

## 6. 数据流

第一版核心数据流：

1. 用户在登录页输入本地账号和密码。
2. 软件校验账号，加载该账号的数据空间。
3. 用户新增影片并输入番号，软件按影片资料匹配顺序尝试匹配封面和基础资料。
4. 软件展示影片匹配候选，用户确认候选、跳过匹配或手动编辑。
5. 用户选择影片类型：单人作品、多人作品或素人。
6. 如果影片是单人作品，用户必须新增或选择女优并输入女优名。
7. 如果影片是多人作品，软件从演员名单中匹配已收藏女优并自动关联；未收藏女优不自动创建，保留演员姓名文本供后续补关联。
8. 后续新增女优时，如果匹配到既有多人作品演员名单，软件弹出补关联确认；用户确认后才建立关联。
9. 如果影片是素人，不强制填写女优名，但允许用户手动关联已收藏女优。
10. 单人作品或需要补充女优资料时，软件按女优资料匹配顺序匹配头像和基础资料，并展示候选。
11. 用户新增或编辑影片、女优、影评、标签和 `[01:23:45]` 格式时间戳。
12. 软件将编辑结果自动保存到本地数据存储。
13. 用户通过影片详情、女优详情和首页入口查看数据。
14. 用户可导出或导入本地数据，导出时默认普通导出，可选择加密导出。
15. 删除影片或女优前需要二次确认；删除女优不删除影片。
16. 数据默认不上传云端。

AI 设置数据流（进入 MVP）：

1. 用户在设置页填写 API Key。
2. 用户填写 Base URL、模型型号 / 模型 ID、供应商备注、温度、最大输出长度、每次推荐参考的女优和影片总数上限等参数；推荐参考总数上限默认值为 30。
3. 用户编辑翻译提示词和首页推荐提示词。
4. 软件将设置保存到当前本地账号的数据空间。
5. 软件重启后可恢复这些设置。

AI 推荐数据流（进入 MVP）：

1. 用户在首页对话框输入偏好描述。
2. 软件先在本地资料库中根据关键词、标签、作品类型、女优、年龄标签、罩杯等条件检索候选影片和女优。
3. 软件读取设置中的“每次推荐参考的女优和影片总数上限”，默认值为 30；该值只能为 0-999；0 表示不限数量。
4. 如果上限大于 0，软件按本地匹配度排序后截取影片和女优总数不超过该上限的数据；如果上限为 0，则发送全部本地候选。
5. 软件发送候选影片/女优的元数据和用户写的文本全文，包括完整影评和完整女优评价。
6. 发送内容不得包含本地账号密码或密码哈希、API Key、数据库路径、本地文件绝对路径或未成年人相关内容。
7. 软件调用大模型 API。
8. 软件展示推荐影片、推荐女优和推荐理由。

AI 翻译数据流（进入 MVP）：

1. 用户对影片标题或简介发起中文化处理。
2. 软件根据翻译提示词和待翻译文本构造请求。
3. 软件调用大模型 API。
4. 软件将翻译结果写入中文标题或中文简介字段，原文保留在原始字段。
5. 已翻译且未被用户要求重新翻译的内容不重复调用大模型。
6. 用户点击 `重新翻译` 时，软件重新调用大模型并覆盖当前中文标题和中文简介，但不删除原始字段。
7. 用户点击 `取消翻译` 时，第一版清空已保存的中文标题和中文简介，回到显示原始标题和原始简介；不需要中断正在进行的请求，但请求返回后不得把已取消的结果重新写入当前影片记录。

调用大模型 API 时需要明确发送范围：推荐发送用户输入、候选影片/女优元数据和用户写的文本全文；翻译发送标题和简介原文。

## 7. 状态管理

- 软件有本地登录状态。
- 软件至少包含登录/注册、首页、影片列表、影片详情、女优列表、女优详情、设置页面。
- 需要保存用户设置，尤其是 API Key、Base URL、模型型号 / 模型 ID、模型参数、推荐参考总数上限、提示词和数据相关配置。
- 需要保存历史记录性质的数据，包括影片影评撰写时间、记录创建时间和更新时间。
- 撤销 / 重做指用户编辑字段、影评、标签后，可以回退上一步或恢复下一步的编辑历史。该能力会增加状态管理复杂度，第一版不做全局撤销 / 重做。
- 草稿指编辑内容先临时保存为未正式提交状态，用户稍后再确认保存。由于 Nvy 第一版已要求自动保存，第一版不做独立草稿系统。
- 不要求实时多人同步。

### 7.1 UI 框架与视觉系统

第一版 UI 不应做成传统资料管理系统或老式 Windows 表单风格。页面视觉和交互要求以 `docs/08_page_wireframes.md` 为准，当前方向为：

1. 以 Google Gemini 和 Google AI Studio 暗色模式为主要参考。
2. 背景使用暗灰色，主色为淡暗紫色和淡暗粉色，允许柔和渐变。
3. 不确定的组件细节优先按 Material Design 3 处理。
4. 全局应用外壳需要包含窄左侧图标栏、Logo、页面内容区和必要的悬浮提示。
5. 左侧栏、搜索栏、下拉栏、按钮、输入框、滚动遮罩、网格卡片、轮播区域应沉淀为可复用 UI 组件。
6. 业务逻辑不应直接写进纯视觉组件；UI 组件通过明确 props 接收数据和回调。
7. 页面级组件负责组合布局和调度状态，搜索解析、标签归一化、姓名匹配、年龄标签计算等核心规则应放在独立逻辑模块。
8. 女优库和影片库采用连续滚动，不做分页。
9. 首页轮播需要支持中心项放大、越远离中央越小的视觉规则。
10. 设置页中“标签库管理”作为独立分区。
11. 详情页 `删除` 不固定在右上角，只在滚动到顶层时可见。

## 8. 数据存储架构

第一版需要本地持久化存储。

优先评估方案：

- 本地 SQLite：保存账号、影片、女优、影评、关联关系、设置；第一版不整体加密数据库。
- 标签库表：保存影片标签和女优标签的规范名、同义词、繁体写法、关联标签和预置来源。
- 设置表：保存大模型 API、模型参数、翻译提示词、首页推荐提示词和启用开关。
- API Key 加密存储：保存加密后的 API Key，不保存明文。
- 本地文件夹：保存封面、头像、导出备份文件和本地图片缓存。
- 可选加密导出文件：用于保护备份和迁移文件。
- SQLCipher 或其他整体加密方案：后续如需要整体加密数据库，可以作为 SQLite 的迁移候选。

推荐本地文件结构：

```text
Nvy/
  data/
    nvy.sqlite
  secrets/
    stronghold.dat
  cache/
    covers/
      {video_id}.{ext}
    actresses/
      {actress_id}.{ext}
    tmp/
  exports/
```

规则：

- 数据库保存结构化数据和相对资源路径。
- 影片封面使用 `covers/{video_id}.{ext}`。
- 女优头像使用 `actresses/{actress_id}.{ext}`。
- 不使用女优名、番号或标题作为主文件名，避免重名、改名和隐私泄露。
- API Key 存入 Tauri Stronghold，不作为普通设置写入 SQLite。

推荐导出包结构：

```text
NvyExport-{username_or_account_id}-{yyyyMMdd-HHmmss}.nvyzip
  manifest.json
  data.json
  assets/
    covers/
    actresses/
  checksums.json
```

- 普通导出扩展名：`.nvyzip`。
- 加密导出扩展名：`.nvyenc`。
- `data.json` 不包含明文 API Key。
- 导入时先校验 manifest 和 checksums，失败时不得破坏现有数据库。

暂不采用：

- 云数据库：第一版不做在线同步。
- 浏览器 LocalStorage：当前目标是独立桌面软件。

## 9. 外部依赖

- 系统文件能力：用于本地图片、数据文件、导入导出。
- 元数据来源：gfriends/gfriends、Yuukiy/JavSP、conanak99/jav-rest-api 暂作为参考和候选来源。
- 已确认来源顺序：女优头像优先 gfriends；女优资料查 DMM/FANZA API；影片资料先观察 JavSP 多站点逻辑，查不到再查 DMM/FANZA API；后续保留影片资料优先 DMM/FANZA API 的可能。
- 中文化处理来源：第一版使用大模型自动翻译标题和简介，用户可手动编辑兜底。
- AI 模型 API：第一版用于首页推荐和标题/简介自动翻译；需要支持 OpenAI Responses API、OpenAI 兼容接口和自定义接口。
- 中文维基百科链接：作为用户保存的外部链接，不一定需要自动调用 API。
- MissAV：用户使用的外部观看网站，第一版不做程序级集成。

## 10. 安全架构

- 有本地用户登录。
- 处理成人内容相关私人记录，隐私敏感。
- 第一版没有管理员权限。
- 密码不应明文保存。
- 第一版本地数据库不整体加密，后续保留整体加密升级可能。
- 导出文件支持可选加密。
- 封面和头像匹配成功后缓存到本地，导入导出需要覆盖图片资源。
- 元数据匹配会把番号或女优名发送到第三方来源，需要明确告知用户。
- 大模型 API 调用后续需要明确发送数据范围，避免意外泄露私人记录。
- API Key 不应写入日志或代码仓库；第一版需要本地加密保存。
- API Key 需要本地加密保存。

## 11. 可扩展性考虑

当前不为远期复杂功能进行过度设计。

可以保留的扩展点：

- 数据存储层可替换。
- 核心业务逻辑与 UI 分离。
- 外部 API 调用集中管理。
- 配置项集中管理。
- 重要数据结构文档化。
- 本地账号和未来云端账号之间保留迁移空间。
- 离线模式和未来在线模式之间保留清晰边界。

## 12. 暂不采用的架构

当前暂不采用：

| 架构            | 暂不采用原因                 |
| --------------- | ---------------------------- |
| 微服务          | 对 MVP 过度复杂              |
| Kubernetes 部署 | 对个人项目和 MVP 过度复杂    |
| 多仓库架构      | 增加维护成本                 |
| 复杂权限系统    | 第一版只有本地普通用户       |
| 事件驱动架构    | 当前需求不明确，可能过度设计 |
| 云端同步架构    | 用户已明确第一版先做离线本地 |
