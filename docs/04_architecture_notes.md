## 3. 架构设计原则

1. 先满足 MVP，不做过度设计。
2. 模块职责清晰。
3. 数据流尽量简单。
4. 优先让项目创建者能理解。
5. 优先让 Coding Agent 能稳定维护。
6. 避免隐藏复杂度。
7. 避免为了未来可能性牺牲当前简单性。

## 4. 可能的架构形态

### 4.1 本地单体应用

适合：

- 个人工具。
- 不需要账号。
- 不需要多人协作。
- 数据保存在本地。
- 需要访问本地文件或硬件。

可能结构：

```text
应用界面
  ↓
业务逻辑
  ↓
本地数据 / 文件 / 系统能力
```

优点：

- 简单。
- 易启动。
- 不需要服务器。
- 适合 MVP。

缺点：

- 多设备同步困难。
- 多人协作困难。
- 发布和升级需要额外考虑。

当前判断：待评估。

### 4.2 Web 前后端一体应用

适合：

- 需要浏览器访问。
- 未来可能部署到云端。
- 需要账号系统。
- 需要数据库。
- 需要跨设备访问。

可能结构：

```
浏览器界面
  ↓
应用服务
  ↓
数据库 / 文件存储 / 第三方 API
```

优点：

- 跨平台。
- 易分享。
- 适合多人访问。

缺点：

- 需要部署。
- 需要考虑后端。
- 需要考虑账号、安全、数据库。

当前判断：待评估。

### 4.3 前后端分离架构

适合：

- 中大型项目。
- 多客户端。
- 复杂 API。
- 明确需要扩展。

可能结构：

```
前端应用
  ↓
后端 API
  ↓
数据库 / 第三方服务
```

优点：

- 边界清晰。
- 可扩展性强。

缺点：

- 对 MVP 可能过度复杂。
- 对初学者学习成本较高。
- 需要维护更多工程。

当前判断：暂不优先，除非需求证明必要。
