# 04 架构说明

## 1. 文档目的

本文档用于记录项目的架构设计、模块划分、数据流、外部依赖和未来扩展方向。

当前项目仍处于需求访谈阶段，因此本文档只记录初步架构假设，不作为最终实现依据。

## 2. 当前架构状态

当前状态：第一版倾向本地单体桌面应用。

原因：

- 软件形态已初步确认为 Windows 独立桌面软件。
- 技术栈尚未确定。
- 核心功能已明确为本地账号、影片记录、女优记录、影评、时间戳、自动保存、数据导入导出、影片元数据匹配和女优元数据匹配。
- 数据存储已确认需要本地持久化；第一版优先 SQLite，不整体加密数据库，使用密码哈希和可选加密导出；匹配到的封面和头像缓存到本地。
- 第一版暂不实现服务器同步，也不实现大模型实际推荐；后续可能需要在线模式和大模型 API。

## 3. 架构设计原则

1. 先满足 MVP，不做过度设计。
2. 模块职责清晰。
3. 数据流尽量简单。
4. 优先让项目创建者能理解。
5. 优先让 Coding Agent 能稳定维护。
6. 避免隐藏复杂度。
7. 避免为了未来可能性牺牲当前简单性。

## 4. 可能的架构形态

### 4.1 本地单体应用

适合：

- 个人工具。
- 需要本地账号。
- 不需要多人协作。
- 数据保存在本地。
- 需要访问本地文件、图片资源或本地数据库。

可能结构：

```text
应用界面
  ↓
业务逻辑
  ↓
本地账号 / 本地数据库 / 本地图片文件 / 设置
```

优点：

- 简单。
- 易启动。
- 不需要服务器。
- 适合 MVP。

缺点：

- 多设备同步困难。
- 多人协作困难。
- 发布和升级需要额外考虑。

当前判断：第一版优先采用该架构方向。

### 4.2 Web 前后端一体应用

适合：

- 需要浏览器访问。
- 未来可能部署到云端。
- 需要账号系统。
- 需要数据库。
- 需要跨设备访问。

可能结构：

```
浏览器界面
  ↓
应用服务
  ↓
数据库 / 文件存储 / 第三方 API
```

优点：

- 跨平台。
- 易分享。
- 适合多人访问。

缺点：

- 需要部署。
- 需要考虑后端。
- 需要考虑账号、安全、数据库。

当前判断：后续在线同步版本可评估，第一版暂不采用。

### 4.3 前后端分离架构

适合：

- 中大型项目。
- 多客户端。
- 复杂 API。
- 明确需要扩展。

可能结构：

```
前端应用
  ↓
后端 API
  ↓
数据库 / 第三方服务
```

优点：

- 边界清晰。
- 可扩展性强。

缺点：

- 对 MVP 可能过度复杂。
- 对初学者学习成本较高。
- 需要维护更多工程。

当前判断：暂不优先，除非需求证明必要。

## 6. 数据流

第一版核心数据流：

1. 用户在登录页输入本地账号和密码。
2. 软件校验账号，加载该账号的数据空间。
3. 用户新增影片并输入番号，软件按影片资料匹配顺序尝试匹配封面和基础资料。
4. 软件展示影片匹配候选，用户确认候选、跳过匹配或手动编辑。
5. 用户选择影片类型：单人作品或多人作品。
6. 如果影片是单人作品，用户新增女优并输入女优名；如果影片是多人作品，软件禁止添加或关联女优。
7. 单人作品场景下，软件按女优资料匹配顺序匹配头像和基础资料，并展示候选。
8. 用户新增或编辑影片、女优、影评和 `[01:23:45]` 格式时间戳。
9. 软件将编辑结果自动保存到本地数据存储。
10. 用户通过影片详情、女优详情和首页入口查看数据。
11. 用户可导出或导入本地数据，导出时默认普通导出，可选择加密导出。
12. 删除影片或女优前需要二次确认；删除女优不删除影片。
13. 数据默认不上传云端。

AI 推荐数据流（后续版本，不进入 MVP）：

1. 用户在设置页填写 API Key。
2. 用户在首页对话框输入偏好描述。
3. 软件从本地资料库取出必要摘要。
4. 软件调用大模型 API。
5. 软件展示候选影片或女优。

需要特别确认：调用大模型 API 时，哪些本地私人数据允许发送给第三方服务。

## 7. 状态管理

- 软件有本地登录状态。
- 软件至少包含登录/注册、首页、影片列表、影片详情、女优列表、女优详情、设置页面。
- 需要保存用户设置，尤其是 API Key 和数据相关配置。
- 需要保存历史记录性质的数据，包括影片影评撰写时间、记录创建时间和更新时间。
- 是否需要撤销 / 重做待确认。
- 是否需要草稿待确认；自动保存可能降低草稿需求。
- 不要求实时多人同步。

## 8. 数据存储架构

第一版需要本地持久化存储。

优先评估方案：

- 本地 SQLite：保存账号、影片、女优、影评、关联关系、设置；第一版不整体加密数据库。
- 本地文件夹：保存封面、头像、导出备份文件和本地图片缓存。
- 可选加密导出文件：用于保护备份和迁移文件。
- SQLCipher 或其他整体加密方案：后续如需要整体加密数据库，可以作为 SQLite 的迁移候选。

暂不采用：

- 云数据库：第一版不做在线同步。
- 浏览器 LocalStorage：当前目标是独立桌面软件。

## 9. 外部依赖

- 系统文件能力：用于本地图片、数据文件、导入导出。
- 元数据来源：gfriends/gfriends、Yuukiy/JavSP、conanak99/jav-rest-api 暂作为参考和候选来源。
- 已确认来源顺序：女优头像优先 gfriends；女优资料查 DMM/FANZA API；影片资料先观察 JavSP 多站点逻辑，查不到再查 DMM/FANZA API；后续保留影片资料优先 DMM/FANZA API 的可能。
- AI 模型 API：用于后续自然语言推荐，不进入第一版 MVP。
- 中文维基百科链接：作为用户保存的外部链接，不一定需要自动调用 API。
- MissAV：用户使用的外部观看网站，第一版不做程序级集成。

## 10. 安全架构

- 有本地用户登录。
- 处理成人内容相关私人记录，隐私敏感。
- 第一版没有管理员权限。
- 密码不应明文保存。
- 第一版本地数据库不整体加密，后续保留整体加密升级可能。
- 导出文件支持可选加密。
- 封面和头像匹配成功后缓存到本地，导入导出需要覆盖图片资源。
- 元数据匹配会把番号或女优名发送到第三方来源，需要明确告知用户。
- 大模型 API 调用后续需要明确发送数据范围，避免意外泄露私人记录。

## 11. 可扩展性考虑

当前不为远期复杂功能进行过度设计。

可以保留的扩展点：

- 数据存储层可替换。
- 核心业务逻辑与 UI 分离。
- 外部 API 调用集中管理。
- 配置项集中管理。
- 重要数据结构文档化。
- 本地账号和未来云端账号之间保留迁移空间。
- 离线模式和未来在线模式之间保留清晰边界。

## 12. 暂不采用的架构

当前暂不采用：

| 架构            | 暂不采用原因                 |
| --------------- | ---------------------------- |
| 微服务          | 对 MVP 过度复杂              |
| Kubernetes 部署 | 对个人项目和 MVP 过度复杂    |
| 多仓库架构      | 增加维护成本                 |
| 复杂权限系统    | 第一版只有本地普通用户       |
| 事件驱动架构    | 当前需求不明确，可能过度设计 |
| 云端同步架构    | 用户已明确第一版先做离线本地 |
