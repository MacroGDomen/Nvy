# 07 决策记录

## 1. 文档目的

本文档用于记录项目中的重要决策。

它的作用是防止项目在多轮对话、多次修改、多次换 Agent 后失去方向。

重要原则：

- 不只记录做了什么，还要记录为什么这么做。
- 不只记录选择了什么，还要记录放弃了什么。
- 不只记录结论，还要记录影响和风险。

## 2. 需要记录的决策类型

以下情况必须记录：

1. 确定项目方向。
2. 确定目标用户。
3. 确定 MVP 范围。
4. 确定技术栈。
5. 改变技术栈。
6. 引入新依赖。
7. 放弃某个功能。
8. 推迟某个功能。
9. 改变数据存储方式。
10. 改变架构方向。
11. 改变部署方式。
12. 改变验收标准。

## 3. 决策记录格式

每条决策使用以下格式：

```markdown
## D001 决策标题

- 日期：
- 状态：已确认 / 待确认 / 已废弃
- 决策：
- 背景：
- 选择理由：
- 备选方案：
- 放弃备选方案的原因：
- 影响：
- 风险：
- 后续行动：
```

## 4. 当前决策记录

## D001 采用文档先行的开发方式

- 日期：YYYY-MM-DD
- 状态：已确认
- 决策：本项目先通过对话理解需求，并维护项目文档；在用户明确确认前，不进入正式编码阶段。
- 背景：项目创建者有软件想法，并希望通过 Codex 等 Coding Agent 协助开发，但目前不熟悉传统软件开发流程。
- 选择理由：
  - 文档可以沉淀对话中的重要信息。
  - 文档可以减少 Agent 遗忘上下文的问题。
  - 文档可以避免过早写代码。
  - 文档可以帮助后续拆分开发任务。
- 备选方案：
  - 直接让 Agent 写代码。
  - 先搭建完整工程模板。
  - 先学习完整软件开发课程。
- 放弃备选方案的原因：
  - 直接写代码容易导致需求不清、返工严重。
  - 先搭建完整工程可能过度设计。
  - 先系统学习完整软件开发周期太长，不适合当前目标。
- 影响：
  - 项目前期会花更多时间在需求和文档上。
  - 后续开发会更稳定。
  - Agent 的工作边界会更清楚。
- 风险：
  - 如果文档长期不更新，仍然会和真实需求脱节。
  - 如果文档写得过度复杂，可能拖慢开发节奏。
- 后续行动：
  - 创建基础文档结构。
  - 通过访谈补充 `docs/02_product_spec.md`。
  - 收敛 MVP 范围。
  - 再进入技术栈选择和开发计划。

## D002 将 `docs/02_product_spec.md` 作为最重要的需求来源

- 日期：YYYY-MM-DD
- 状态：已确认
- 决策：将 `docs/02_product_spec.md` 作为项目最重要的软件说明文档。
- 背景：项目需要让 Coding Agent 长期理解和维护用户的软件意图。
- 选择理由：
  - 软件说明文档可以统一记录目标、用户、功能、流程、数据、验收标准。
  - 后续开发任务可以从该文档拆解。
  - 发生需求冲突时，可以优先参考该文档。
- 备选方案：
  - 只依赖聊天上下文。
  - 只写 README。
  - 只写功能列表。
- 放弃备选方案的原因：
  - 聊天上下文容易丢失和混乱。
  - README 更适合对外说明，不适合承载完整需求。
  - 功能列表无法表达用户场景、边界、数据和验收标准。
- 影响：
  - 每次重要需求变化都需要同步更新该文档。
  - 代码实现前需要先检查该文档。
- 风险：
  - 文档过长后可能维护成本上升。
- 后续行动：
  - 使用模板初始化 `docs/02_product_spec.md`。
  - 每轮访谈后更新该文档。
  - 在进入开发前审查该文档是否足够明确。
