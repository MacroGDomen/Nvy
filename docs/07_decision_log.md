# 07 决策记录

## 1. 文档目的

本文档用于记录项目中的重要决策。

它的作用是防止项目在多轮对话、多次修改、多次换 Agent 后失去方向。

重要原则：

- 不只记录做了什么，还要记录为什么这么做。
- 不只记录选择了什么，还要记录放弃了什么。
- 不只记录结论，还要记录影响和风险。

## 2. 需要记录的决策类型

以下情况必须记录：

1. 确定项目方向。
2. 确定目标用户。
3. 确定 MVP 范围。
4. 确定技术栈。
5. 改变技术栈。
6. 引入新依赖。
7. 放弃某个功能。
8. 推迟某个功能。
9. 改变数据存储方式。
10. 改变架构方向。
11. 改变部署方式。
12. 改变验收标准。

## 3. 决策记录格式

每条决策使用以下格式：

```markdown
## D001 决策标题

- 日期：
- 状态：已确认 / 待确认 / 已废弃
- 决策：
- 背景：
- 选择理由：
- 备选方案：
- 放弃备选方案的原因：
- 影响：
- 风险：
- 后续行动：
```

## D009 第一版安全存储策略

- 日期：2026-06-19
- 状态：已确认
- 决策：第一版采用“不整体加密数据库 + 密码哈希 + 可选加密导出”的安全策略，并保留后续升级为整体加密数据库的可能。
- 背景：本项目保存成人内容相关私人记录，数据隐私敏感。用户已明确希望第一版先不做数据库整体加密，但要保留后续整体加密可能。
- 选择理由：
  - 不整体加密数据库可以降低第一版实现、调试、备份和恢复复杂度。
  - 密码哈希可以避免本地账号密码明文保存。
  - 可选加密导出可以保护迁移和备份文件。
  - 保留整体加密扩展点，便于后续在隐私要求提高时迁移。
- 备选方案：
  - 第一版直接使用整体加密数据库。
  - 完全不提供任何加密能力。
- 放弃备选方案的原因：
  - 直接整体加密会引入密钥管理、忘记密码无法恢复、依赖选择和迁移复杂度。
  - 完全不提供加密能力不适合本项目的隐私敏感属性。
- 影响：
  - 第一版数据库文件如果被直接复制，仍有被读取的风险，需要在安全说明中明确。
  - 导出功能需要支持普通导出和加密导出两种路径。
  - 数据结构设计应避免绑定死具体数据库实现，以便后续整体加密迁移。
- 风险：
  - 用户可能误以为本地登录等于数据库文件安全。
  - 加密导出密码丢失后，导出文件可能无法恢复。
- 后续行动：
  - 在技术方案阶段选择密码哈希算法和导出加密方式。
  - 设计数据层时保留 SQLCipher 或其他整体加密方案的迁移空间。

  - 多人作品不添加女优可以避免自动建立不准确的女优关联。
  - 先观察 JavSP 多站点逻辑质量，可以基于实际效果决定是否调整影片资料来源。
- 备选方案：
  - 时间戳使用无括号格式。
  - 删除时不二次确认。
  - 多人作品也自动添加或关联女优。
  - 影片资料直接优先 DMM/FANZA API。
- 放弃备选方案的原因：
  - 无括号时间戳可读性和识别性较弱。
  - 不二次确认会增加误删风险。
  - 多人作品自动关联女优可能产生错误数据。
  - 影片资料来源优先级需要先观察 JavSP 逻辑质量再决定是否调整。
- 影响：
  - 数据模型需要支持女优字段：名字、头像、三围、生日、身高、维基百科链接、个人评价、罩杯、出道日期。
  - 数据模型需要支持影片字段：番号、标题、封面、女优、发行日期、片长、来源链接、影评、简介、作品类型、标签。
  - 影片与女优关联逻辑需要根据作品类型分支。
  - 删除流程需要二次确认。
- 风险：
  - 多人作品不关联女优会降低从女优维度回顾多人作品的能力，但符合当前用户要求。
  - JavSP 多站点逻辑可能受站点变化影响，需要保留 DMM/FANZA API fallback。
- 后续行动：
  - 在产品规格和验收中补充字段与规则。
  - 技术方案阶段评估 JavSP 逻辑质量和 DMM/FANZA API 可用性。
