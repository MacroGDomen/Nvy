# 07 决策记录

## 1. 文档目的

本文档用于记录项目中的重要决策。

它的作用是防止项目在多轮对话、多次修改、多次换 Agent 后失去方向。

重要原则：

- 不只记录做了什么，还要记录为什么这么做。
- 不只记录选择了什么，还要记录放弃了什么。
- 不只记录结论，还要记录影响和风险。

## 2. 需要记录的决策类型

以下情况必须记录：

1. 确定项目方向。
2. 确定目标用户。
3. 确定 MVP 范围。
4. 确定技术栈。
5. 改变技术栈。
6. 引入新依赖。
7. 放弃某个功能。
8. 推迟某个功能。
9. 改变数据存储方式。
10. 改变架构方向。
11. 改变部署方式。
12. 改变验收标准。

## 3. 决策记录格式

每条决策使用以下格式：

```markdown
## D001 决策标题

- 日期：
- 状态：已确认 / 待确认 / 已废弃
- 决策：
- 背景：
- 选择理由：
- 备选方案：
- 放弃备选方案的原因：
- 影响：
- 风险：
- 后续行动：
```

## D009 第一版安全存储策略

- 日期：2026-06-19
- 状态：已确认
- 决策：第一版采用“不整体加密数据库 + 密码哈希 + 可选加密导出”的安全策略，并保留后续升级为整体加密数据库的可能。
- 背景：本项目保存成人内容相关私人记录，数据隐私敏感。用户已明确希望第一版先不做数据库整体加密，但要保留后续整体加密可能。
- 选择理由：
  - 不整体加密数据库可以降低第一版实现、调试、备份和恢复复杂度。
  - 密码哈希可以避免本地账号密码明文保存。
  - 可选加密导出可以保护迁移和备份文件。
  - 保留整体加密扩展点，便于后续在隐私要求提高时迁移。
- 备选方案：
  - 第一版直接使用整体加密数据库。
  - 完全不提供任何加密能力。
- 放弃备选方案的原因：
  - 直接整体加密会引入密钥管理、忘记密码无法恢复、依赖选择和迁移复杂度。
  - 完全不提供加密能力不适合本项目的隐私敏感属性。
- 影响：
  - 第一版数据库文件如果被直接复制，仍有被读取的风险，需要在安全说明中明确。
  - 导出功能需要支持普通导出和加密导出两种路径。
  - 数据结构设计应避免绑定死具体数据库实现，以便后续整体加密迁移。
- 风险：
  - 用户可能误以为本地登录等于数据库文件安全。
  - 加密导出密码丢失后，导出文件可能无法恢复。
- 后续行动：
  - 在技术方案阶段选择密码哈希算法和导出加密方式。
  - 设计数据层时保留 SQLCipher 或其他整体加密方案的迁移空间。

## D010 第一版资源缓存、导出默认行为和发布范围

- 日期：2026-06-19
- 状态：已确认
- 决策：匹配成功后的影片封面和女优头像缓存到本地；导出默认普通导出，加密导出作为可选项；首页轮播展示所有已记录影片和女优，加入软件即视为收藏；第一版仅供项目创建者本人使用，稳定后再考虑安装包。
- 背景：用户回答了关于图片保存、导出默认行为、首页收藏定义和第一版发布方式的问题。
- 选择理由：
  - 本地缓存图片可以减少外部图片链接失效对资料库浏览的影响。
  - 默认普通导出降低自用场景下的备份操作成本，同时保留加密导出给隐私更敏感的场景。
  - 将所有已记录影片和女优视为收藏，可以避免第一版额外设计收藏状态。
  - 第一版自用可以暂缓安装包、自动更新和发布流程，降低 MVP 复杂度。
- 备选方案：
  - 只保存外部图片地址。
  - 默认加密导出。
  - 单独设计手动收藏功能。
  - 第一版直接打包安装包发布。
- 放弃备选方案的原因：
  - 外部图片地址容易失效。
  - 默认加密导出会增加第一版备份恢复的操作成本。
  - 手动收藏会增加状态和交互复杂度。
  - 安装包发布不是第一版验证核心价值所必需。
- 影响：
  - 导入导出需要覆盖图片缓存资源。
  - 首页轮播无需单独收藏表或收藏字段即可启动。
  - 技术方案阶段仍需决定图片缓存目录、文件命名和导出包格式。
- 风险：
  - 图片缓存会增加本地存储占用。
  - 普通导出文件如果被他人获得，可能暴露隐私数据。
- 后续行动：
  - 设计图片缓存目录和导出包结构。
  - 在界面中清楚区分普通导出和加密导出。

  - Electron + React + TypeScript。

  - 如果采用 Tauri，第一版应通过统一 `desktopApi`、数据访问层和纯 TypeScript 业务模块保留迁移到 Electron 的可能。
  - React + TypeScript 前端、搜索规则、标签规则、姓名归一化、导入导出格式和 SQLite 数据模型应尽量保持框架无关。

## D022 工程实现阶段采用 Vitest 与 rusqlite

- 日期：2026-06-22
- 状态：已确认
- 决策：第一版工程实现中，前端 / TypeScript 纯逻辑单元测试采用 Vitest；Tauri / Rust 侧 SQLite 访问采用 `rusqlite`，并启用 `bundled` 特性。Playwright 仍作为后续端到端测试候选，暂未接入。

- 选择理由：
  - Vitest 与 Vite / TypeScript 项目集成成本低，适合先覆盖搜索、标签、姓名归一化等纯逻辑模块。
  - `rusqlite` 是 Rust 生态成熟 SQLite 访问库，适合 Tauri 命令侧维护本地数据库初始化和迁移。
  - `rusqlite` 的 `bundled` 特性可以降低 Windows 上系统 SQLite 依赖不一致的问题。
- 备选方案：
  - Jest。
  - 只做手动测试。
  - Tauri SQL 插件。
  - 前端直接访问 SQLite。
- 放弃或暂缓备选方案的原因：
  - Jest 在 Vite 项目中需要更多配置，当前没有明显收益。
  - 只做手动测试不利于后续复杂搜索和归一化规则维护。
  - Tauri SQL 插件可以后续评估，但当前 Rust 侧直接管理迁移更直观。
  - 前端直接访问 SQLite 会破坏 `desktopApi` 边界，不利于迁移和维护。
- 影响：
  - `package.json` 新增 `test` 脚本和 `vitest` 开发依赖。
  - `src-tauri/Cargo.toml` 新增 `rusqlite` 依赖。
  - 数据库初始化和迁移逻辑放在 `src-tauri/src/db/`。
  - 前端通过 `src/services/desktopApi/` 调用 Tauri 命令，不直接依赖 SQLite。
- 风险：
  - `rusqlite` bundled 编译会增加首次 Rust 构建时间。
  - 后续如果改用 SQLCipher 或整体数据库加密，需要迁移 SQLite 访问层。
- 后续行动：
  - 后续实现账号、资料库和设置时继续沿用 repository / service 边界。
  - 在搜索、标签、姓名归一化规则出现时优先补充 Vitest 单元测试。

## D023 - 第一版本地账号密码哈希采用 Argon2

- 日期：2026-06-22
- 状态：已确认并已实现
- 决策：第一版 Nvy 的本地账号密码使用 Rust 侧 `argon2` 哈希保存，配合 `rand_core` 生成随机盐，账号 id 使用 `uuid` 生成。
- 原因：
  1. 已确认需求要求密码哈希保存，不能明文保存。
  2. Argon2 是成熟密码哈希方案，避免自制密码存储逻辑。
  3. 本地账号只用于本机数据隔离，不引入云端账号或服务器同步。
- 影响范围：
  1. `src-tauri/src/auth/mod.rs` 负责注册、登录、密码校验和账号数据隔离基础查询。
  2. 前端通过 `src/services/desktopApi` 调用账号相关 Tauri command，不直接处理哈希。
  3. 后续导出、日志和错误处理不得暴露密码或密码哈希。
- 备选方案：
  1. 普通 SHA 系列摘要：暂不选择，因为不适合作为密码存储默认方案。
  2. bcrypt：成熟可用，但第一版先采用 Argon2，避免同时引入多套密码哈希策略。
  3. 明文保存：不符合需求，明确排除。
