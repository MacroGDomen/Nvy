# 00 项目简述

## 1. 项目名称

Nvy。

## 2. 一句话描述

这是一个面向个人成人影片观看者的 Windows 本地桌面软件，用于记录成人影片、女优资料、观影影评和带时间点的个人感受，并通过用户自填 API 接入大模型进行本地资料库内的检索推荐和标题/简介翻译。

## 3. 项目背景

项目创建者在浏览和观看成人影片时，会对某些影片、女优或影片中的具体时间点产生个人评价和观后记录需求。现有成人视频网站主要提供观看、收藏和浏览能力，但缺少私人化、结构化的影评记录、时间点标注、女优评价和跨影片回顾能力。

用户希望在以后查看自己的收藏或记录时，能够知道自己什么时候看过某部影片、当时的评价是什么、哪些时间点值得回看，以及某位女优相关影片和个人评价。

## 4. 核心问题

本项目试图解决的核心问题是：

成人影片观看后的私人记录容易遗忘，现有网站收藏夹无法承载影片影评、具体时间点感受、女优评价和本地化可检索资料库。

## 5. 目标用户

详见 `docs/01_user_profile.md`。

当前简述：

核心目标用户是项目创建者本人，以及具有类似需求的成年个人用户。该用户在 Windows 电脑上观看成人影片，希望用本地工具保存私人观影记录。

## 6. 第一版目标

第一版 MVP 的目标是：

在 Windows 本地离线模式下，完成“本地账号登录 -> 输入番号或女优名匹配基础资料 -> 缓存封面和头像 -> 用户自由修改影片和女优资料 -> 区分单人 / 多人作品 -> 撰写影评并插入 `[01:23:45]` 格式时间戳 -> 自动保存 -> 导入导出 -> 首页轮播和后续查看记录”的核心闭环。

第一版技术栈已确认：Tauri + React + TypeScript + SQLite。

第一版 6 个核心页面低保真草图已创建，详见 `docs/08_page_wireframes.md`。

第一版 UI 方向已调整为参考 Google Gemini / Google AI Studio 暗夜模式，采用暗灰背景、淡暗紫和淡暗粉强调色、柔和渐变和 Material Design 3 倾向。

MVP 应该尽量小，只验证软件的核心记录价值，不追求完整在线同步、商业化发布或复杂多轮推荐能力。

## 7. 第一版边界

### 7.1 第一版必须解决

- 本地注册和登录。
- 多个本地账号之间的数据隔离。
- 影片资料记录，包括番号、名称、封面和影评。
- 影片资料字段包括番号、标题、封面、女优、发行日期、片长、来源链接、影评、简介、作品类型和标签。
- 输入番号后自动匹配影片封面和基础资料，匹配结果允许用户自由修改。
- 匹配成功后将影片封面缓存到本地。
- 影评撰写时间记录。
- 影评编辑时支持通过时分秒输入快速插入时间戳。
- 女优资料记录，包括名字、头像、三围、中文维基百科链接和个人评价。
- 女优资料字段包括名字、头像、三围、生日、身高、维基百科链接、个人评价、罩杯和出道日期。
- 输入女优名后自动匹配头像和女优基础资料，匹配结果允许用户自由修改。
- 匹配成功后将女优头像缓存到本地。
- 能够从女优查看软件中已记录的相关影片。
- 首页轮播展示所有已记录影片和女优，加入软件即视为收藏。
- 作品类型包含单人作品、多人作品、素人；番号和作品类型必填，单人作品女优必填。
- 多人作品只自动关联已收藏女优，未收藏女优不自动创建，后续添加匹配女优时弹出确认后补关联。
- 素人作品允许手动关联已收藏女优。
- 删除影片或女优需要二次确认，确认后永久删除；删除女优不删除影片。
- 支持影片和女优独立标签库、预置标签、组合搜索、同义/关联标签检索、筛选和排序。
- 影片标题和简介优先中文显示，第一版使用大模型自动翻译并缓存翻译结果。
- 设置页保留大模型 API、接口类型、Base URL、模型型号 / 模型 ID、模型参数、推荐参考总数上限、翻译提示词和首页推荐提示词配置；API Key 本地加密保存。
- 首页下方文本框调用大模型与用户交互，检索本地数据库并推荐影片和女优；推荐默认启用。
- 启用翻译后，调用大模型自动翻译影片标题和简介，并将翻译结果保存到本地，避免重复翻译。
- 本地自动保存，不要求用户手动保存。
- 数据导入导出，默认普通导出，加密导出作为可选项。

### 7.2 第一版暂不解决

- 在线服务器同步。
- 云端账号体系。
- 多设备同步。
- 与 MissAV 深度集成或播放影片。
- 更复杂的大模型推荐策略、多轮记忆和自动调参暂不进入第一版。
- 安装包发布暂不进入第一版，第一版先供项目创建者本人使用。

### 7.3 第一版明确不做

- 不提供成人影片播放能力。
- 不下载、托管或分发成人影片内容。
- 不绕过第三方网站的访问控制。
- 不处理未成年人相关内容。

## 8. 成功标准

第一版可以被认为成功的条件：

- 用户能够在 Windows 本地创建账号并登录。
- 用户能够录入一部影片、一位女优，并建立二者关系。
- 用户只输入番号或女优名时，软件能尝试匹配基础资料；匹配失败时仍允许用户手动创建和编辑。
- 用户能够为影片撰写影评，并插入可读的时间戳。
- 用户能够为女优撰写评价，并查看该女优相关影片。
- 用户能够导入导出本地数据。
- 用户能够看到缓存到本地的封面和头像。
- 用户能够在首页轮播中看到所有已记录影片和女优。
- 用户能够通过标签、关键词、作品类型和排序组合检索影片，并通过多语种姓名、标签和罩杯筛选女优。
- 用户关闭并重新打开软件后，数据仍然存在且归属正确账号。
- 用户不需要手动保存，普通编辑操作会自动持久化。

## 9. 项目约束

当前已知约束：

| 约束类型 | 内容 | 影响 |
|---|---|---|
| 开发者经验 | 项目创建者不熟悉传统软件开发 | 技术栈需要尽量简单，文档需要清晰 |
| 内容性质 | 软件记录成人内容相关的私人数据 | 需要强调本地隐私、安全、合法合规和用户自担内容来源 |
| 平台 | Windows 11 和 Windows 10 电脑端 | 技术栈优先考虑桌面应用 |
| 数据 | 第一版保存在本地 | 需要本地数据库或本地文件方案 |
| 账号 | 第一版需要本地账号制度 | 需要本地认证和账号间数据隔离 |
| 联网 | 第一版主要离线使用，但离线不等于禁止联网 | 元数据匹配、大模型推荐和大模型翻译需要联网 |
| 部署 | 第一版自用，不优先打包安装包，暂不考虑服务器部署 | 在线模式和安装包后续再设计 |
| 预算 | 暂未确定 | AI API 成本和打包发布成本待确认 |
| 资料库规模 | 第一版个人规模资料库支持 1999 条记录 | 影响列表性能、搜索性能和测试数据规模 |
